# Battle UI — парадигмы дизайна

Этот документ описывает визуальный язык боевого фронтенда (`battle/frontend`).
Цель — интерфейс в духе **Baldur's Gate 3**: понятный, фэнтезийный, расширяемый.
Любая будущая правка UI должна ему соответствовать. Бэкенд при этом **не трогаем**.

---

## 1. Базовые принципы

1. **Иконка = действие.** Всё, что можно нажать в бою (действия, заклинания,
   способности), — это квадратная иконка-кнопка в «хотбаре», а не текстовая кнопка
   и не выпадающий список. Пользователь должен «читать» панель глазами, как в BG3.
2. **Состояние читается мгновенно** по виду кнопки (см. §3). Никаких скрытых
   состояний — доступность, выбор и стоимость видны сразу.
3. **Любая кнопка раскрывается в «инспектор»** при наведении (см. §4): большая
   выезжающая иконка + структурированное описание.
4. **Фэнтези, но сдержанно.** Тёмный фон, золото, мягкое свечение на ховере.
   Свечение — только как обратная связь (hover/selected), не как постоянный шум.
5. **Расширяемость.** Новый контент = иконка-файл + запись в данных. Никаких
   правок вёрстки под конкретное заклинание.

---

## 2. Палитра и токены

Все цвета — через CSS-переменные в `src/styles.css` (`:root`). Не хардкодить hex
в компонентах.

- Фон/панели: `--bg`, `--panel`, `--panel-2`; рамки: `--border`, `--gold-dim`, `--gold`.
- Текст: `--text`, `--text-dim`; акцент-золото: `--gold`.
- **Типы урона** — переменные `--dt-*` (necrotic/fire/cold/acid/radiant/psychic/force/physical).
  Цвет урона в тултипе берётся из них. При добавлении нового типа урона:
  добавь переменную в `:root` **и** запись в `DMG_COLOR`/`DMG_GLYPH` в `src/battle/tooltip.ts`.
- Радиусы: кнопки `10px`, тултип `12px`. Шрифт заголовков — `Georgia, serif` (золотой).

---

## 3. Хотбар и кнопки (`Hotbar.tsx`, `.ab` в styles.css)

- Кнопка `.ab`: **56×56**, скруглённый квадрат, «каменная» подложка
  (radial-gradient), рамка.
- Группы по типу ресурса с подписью и цветной точкой:
  `Действия (main)` · `Бонусные (bonus)` · `Реакции (reaction)` ·
  `Заклинания (spell)` · `Пассивные (passive)`. Пустые группы скрываются.
- **Состояния** (класс на `.ab`):
  - `ready` — золотой ободок; на ховере приподнимается + золотое свечение иконки.
  - `selected` — золотое кольцо (ждёт цель на сетке): `box-shadow: 0 0 0 2px gold …`.
  - `disabled` — обесцвечено (grayscale), `cursor: not-allowed`, не кликается.
- **Бейдж** (`.ab-badge`, нижний-правый угол) — заряды/число использований
  или круг слота.
- **Уголок** (`.ab-corner`, верхний-левый) — цветной маркер типа действия
  (`bonus` оранжевый, `reaction` синий), чтобы отличать от основного.
- Иконки: `onError` → всегда фолбэк на `/icons/default.png`. Иконки **никогда**
  не «ломаются» в пустоту.

## 4. Инспектор-тултип (`InspectTooltip.tsx`, `.tip*` в styles.css)

Структура сверху вниз (повторяет скриншоты BG3):

1. **Большая иконка** — выезжает из правого-верхнего угла (`.tip-bigicon`,
   частично за пределами карточки) с золотым свечением; плавно появляется.
2. **Заголовок** (`.tip-title`, Georgia, золотой) + **подтип** (`.tip-subtype`,
   курсив, приглушённый): `«Заговор · Некромантия»`, `«Действие · ближний бой»`,
   `«Бонусное действие · Воин»`, `«Пассивный»`.
3. **Стат-блок** (`.tip-stats`) — **строки с подписями, по одной на строку**:
   - Атака: `Атака:` + зелёный кубик `к20` + бонус (`+ 5`).
   - Спасбросок: `Спасбросок:` + кубик `СБ` + характеристика и `Сл N`.
   - Урон: `Урон:` + кости + **тип урона в именительном падеже**, окрашенный
     по типу (`1к8 · дробящий`, `1к6 · кислота`) со значком-глифом.
   - Лечение: `Лечение:` + кости (золотой).
   > Урон и атака/спас **всегда на разных строках** — это явное требование.
4. **Описание** (`.tip-desc`) — поддерживает `**жирные**` сегменты для подсветки
   ключевых терминов (золотым).
5. **Строка спасброска** (`.tip-saveline`) — «При успехе — половина урона» и т.п.
6. **Meta** (`.tip-meta`) — дальность 🎯 / длительность ⏱ / концентрация ◈ /
   область ⊙. Только релевантное (мгновенные длительности скрываются).
7. **Плашка стоимости** (`.tip-costbar`) — пипсы: `● Действие` (круг, золото),
   `■ Бонусное действие` (квадрат, оранжевый), `◆ Реакция` (ромб, синий),
   `■ Слот N круга` (квадрат, фиолетовый, для заклинаний уровня > 0).

Позиционирование: тултип `position: fixed`, центрируется над кнопкой, при нехватке
места сверху — раскрывается вниз; кламп по краям окна (см. `Hotbar.tsx`).

## 5. Данные и источник истины (`src/battle/tooltip.ts`, `src/data/spellIcons.ts`)

- `TooltipData` — единая модель для действия/заклинания/способности. Любую
  сущность сводим к ней билдером.
- **Заклинания**: `buildSpellTooltip(spell)` берёт данные бэкенда (`Spell`) и
  обогащает их русскоязычным `SPELL_ICONS` (матч по английскому имени `en`).
  EN→RU словари (школа, тип урона, характеристика) — в том же файле.
- **Иконки заклинаний**: `spellIconPath(name, level)` → `/icons/spells/cantrips/<key>.png`,
  ключ из `spellIconKey(en)`. Нет иконки → `default.png`.
- **Действия/способности**: статичные определения в `ACTION_DEFS` (правила-текст,
  не зависят от персонажа), как в BG3.

### Заглушки бэкенда (TODO)

Бонус атаки и Сл спасброска пока считаются **не** на бэкенде. До их появления:

- **Атака** без `spell.attack_bonus` → строка показывает просто `к20` (без бонуса).
- **Спасбросок** без `spell.save_dc` → `Сл 10` (`DEFAULT_SAVE_DC`).

Когда бэкенд начнёт отдавать `attack_bonus` / `save_dc` (см. `TODO(backend)` в
`tooltip.ts` и `client.ts`), билдер подхватит их автоматически — заглушки заменятся
реальными значениями без правок UI.

## 6. Как добавить новый контент

- **Новое заклинание-заговор**: положи иконку `public/icons/spells/cantrips/<key>.png`
  и запись `<key>` в `src/data/spellIcons.ts` (имя, школа, roll, dmg, dtype, desc…).
- **Новое действие/способность**: иконка в `public/icons/actions/...` + запись в
  `ACTION_DEFS` и привязка кнопки в нужной группе хотбара.
- **Новый тип урона**: переменная `--dt-*` в `styles.css` + записи в
  `DMG_COLOR`/`DMG_GLYPH`.
- Никогда не верстай под конкретную сущность — расширяй данные.

## 7. Хранение изображений (нестатичные иконки)

Иконки заклинаний (и в будущем — портреты и т.п.) **не статичны**: они хранятся в
БД и редактируются в рантайме. Статичными остаются только `default.png` и
`icons/actions/*` (правила-действия, не меняются).

**Бэкенд** ([`images_repo.py`](../backend/images_repo.py)) — то же «двойное хранилище»,
что и у `JsonRepo`:
- `DATABASE_URL` задан → таблица `battle_images(id, mime, bytes, …)` (миграция
  `006_images` в [`dbcore.py`](../backend/dbcore.py));
- иначе → файлы `backend/data/images/<key>.<ext>` (локальная разработка без БД).

**Роуты** (в [`main.py`](../backend/main.py)):
- `GET /images/{key}` — отдаёт байты с `Cache-Control: public, max-age=300` + `ETag`
  (по `updated_at`); повторные запросы получают `304`. Подмена картинки меняет
  `updated_at` → ETag → клиент берёт новую версию.
- `POST /images` (multipart, поле `file`, опц. `key`) — создать/заменить, вернёт `{id}`.
- `DELETE /images/{key}`.
- Префикс `/images` добавлен в vite-прокси и [`nginx.conf.template`](nginx.conf.template).
- Нужна зависимость `python-multipart` (в `requirements.txt`).

**Сущность ссылается на картинку** полем `image` (ключ). Резолвер на фронте —
`spellImageUrl(spell)` в [`tooltip.ts`](src/battle/tooltip.ts): `spell.image` →
`/images/{key}`, иначе `default.png`. Соглашение об именах файлов больше **не**
используется для картинок (а `spellIcons.ts` остаётся как текстовые метаданные).

**Загрузка/замена** — в редакторе заклинания ([`SpellEditorPage.tsx`](src/pages/SpellEditorPage.tsx)):
аплоадер переиспользует тот же `key` при замене, чтобы не плодить сирот.

### Сид иконок в новое окружение

Исходник иконок: `backend/seed/spell_icons/<key>.png` + `backend/seed/icon_index.json`
({key → английское имя заклинания}). Чтобы залить их в активное хранилище и
проставить `spell.image`:

```bash
cd battle/backend && python seed_images.py   # идемпотентно; работает и с Postgres, и с файлами
```

Запускать один раз на окружение (локально, и отдельно — на проде/Railway, т.к. там
своя БД). Скрипт сам применяет миграции.
