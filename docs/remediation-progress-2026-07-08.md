## ❌ Что осталось нереализованным в ревью Fable

### Движок P0 (3) — брать в первую очередь, свежей сессией
- **C9** — единый расчёт КЗ + заклинательство подкласса. КЗ считается 3 способами (в БД без брони, на листе с бронёй); чинит null-каст Мист. рыцаря/Ловкача. **Трогает персистимый в БД AC — осторожно.**
- **C2** — эффекты на цель (`who:'target'`). Состояния/урон ложатся на исполнителя, не на цель. Нужна модель состояния цели.
- **C3** — ширина шины событий + `endTurn`. Эмитится 3/22 событий; нет конца хода, тика длительностей, повторных спасбросков. **Keystone** — открывает реакции, тикающие яды/горение, Ready, Recharge.

### Контент/прогрессия P0 (3) — высокая пользовательская ценность (прогрессия сломана)
- **D1** — слоты кастеров + апкаст. Сейчас только `spell_slot_1` → **стена на 3 уровне** для всех заклинателей.
- **D3** — грант-тройка `ability_score`/`sense`/`speed` (валидны в схеме после C13, но игнорируются рантаймом).
- **D2** — ASI / черта на 4 уровне (после D3).

### Движок P1 (8)
- **C4** — очередь событий вместо рекурсии (латентный stack overflow).
- **C5** — алгебра модификаторов: `mode` + `priority` (Foundry-модель; «Пояс силы огра» невыразим).
- **C6** — единый резолвер длительностей (баффы «на N раундов» висят вечно).
- **C7** — свёртка преимущества/помехи (adv+dis должны взаимоуничтожаться).
- **C8** — обобщение парадигмы №3 на speed/initiative + сохранение имён методов.
- **C10** — `target_has_condition` и prone (преимущество по распластанному не работает).
- **C11** — дальнобойное оружие (лук стреляет от СИЛ; `ranged` в движке нет).
- **C14** — ручные броски листа: полный путь модификаторов (двойной счёт литералов).
- **C16** — зоны/эманации/призывы (после C2/C3; упирается в многоакторность).

### Движок P2 (1)
- **C12** — `runAbilityCheck`: гейт по владению + `on_success` через общий `applyPayloads` + `mode:'dc'`.

### Контент P1 (2)
- **D4** — ремеханизация ядровых способностей (после C1/C3/C11).
- **D5** — состояния: полные 15/15 + «сильные» половины (после C3/C5).

### Чистка дед-кода A (8)
- P1: **A1** (git-мусор: бинарь `backend/main`, dump-файлы, tsbuildinfo), **A2** (персонажи v1/v2 на бэке).
- P2: **A4** (мёртвые data-каталоги фронта), **A5** (мёртвые ассеты public), **A6** (мёртвые эндпоинты/функции бэка — вместе с A8), **A7** (легаси-каталоги корня), **A8** (кладбище scripts/ + backend/tests), **A9** (мелкий мёртвый код фронта).

### Унификация F (11) — ускоряет дальнейшую разработку
- P1: **F1** (фабрика API-клиентов), **F2** (единый модуль редкости + живой баг), **F4** (дизайн-токены — **разблокирует весь редизайн**), **F5** (одна BG3-карточка), **F6** (EntityDetailShell для 9 модалок), **F7** (useEntityEditor + CreatorLayout для 11 конструкторов), **F8** (один ховер-примитив — нужен для G1), **F10** (реестр сущностей в CardLibrary), **F11** (Generic-CRUD на Go).

### UX/UI G (10)
- P0: **G1** — тач-протокол превью (после F8): парадигма №2 не работает на планшете/телефоне.

- **B8** (GET characters-v3 пагинация — этап H, auth-смежно), **D6** (кости хитов), **H** (авторизация — до предзапуска), полная мобильная адаптация.
- **B2b** (трим списков CardLibrary/кузницы — нужен рефактор превью на fetch-by-id), **мобильный таб-бар для листа и конструкторов** (сквозной шелл из E6).

## 🗺️ Рекомендуемый порядок дальнейших работ

1. **Завершить движок P0** (свежая сессия): **C9** (самодостаточно) → **C2** → **C3** (keystone).
   C3 разблокирует реакции, тик длительностей, конец хода.
2. **Контент P0 прогрессии**: **D3** (грант-тройка) → **D2** (ASI) и **D1** (слоты кастеров) —
   снимает стену 3 уровня и даёт рост характеристик. Максимальная пользовательская ценность.
3. **Движок P1/P2**: C4 (очередь — сразу после C3), C5, C6, C7, C8, C10, C11, C14, C12; затем C16 (зоны).
4. **Контент P1**: D4, D5 (после соответствующих C-задач).
5. **Чистка A** (перед унификацией): A1, A2, затем A4–A9.
6. **Унификация F**: быстрые F1–F5 (F4 дизайн-токены — раньше, разблокирует редизайн) → каркасы F6–F9 → крупные F10–F11.
7. **UX/UI G**: F8 → **G1** (тач-протокол) → G2–G5 → G6–G10.
8. **Этап H (авторизация)** — последним, перед предзапуском.

### Сквозные принципы (держать при каждой задаче)
- Парадигмы №1 (без хардкода), №2 (превью везде, в т.ч. на тач), №3 (методы расчёта), №4 (гранулярность),
  и **№5** (всё — эффект или переменная; типы сущностей — удобные проекции над эффект-субстратом).
- Движок P0 (C2/C3) проектировать так, чтобы `actorId`/`EncounterState` лёг сверху без переписывания (задел под 2D-бой).
- Каждый «вертикальный срез» закрывать до конца: схема ↔ движок ↔ UI, честная деградация вместо молчания.
