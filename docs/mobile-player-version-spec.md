# Техническое задание: отдельная мобильная версия Bag of Holding

**Статус:** согласовано по результатам продуктового интервью

**Приоритет:** недельный рывок, направление №1

**Основной пользователь:** игрок за игровым столом
**Визуальный референс внутри проекта:** `/characters-v3/:id`

## 1. Цель

Создать отдельную мобильную версию сервиса для телефонов и планшетов. Она должна в первую очередь позволять игроку:

1. открыть список своих персонажей;
2. создать персонажа первого уровня;
3. использовать лист персонажа за игровым столом;
4. повышать уровень персонажа;
5. добавлять готовые сущности библиотеки в лист персонажа.

Мобильная версия создаётся с нуля как отдельный UI-контур, а не как адаптивная модификация существующего листа.

При этом мобильный и десктопный интерфейсы обязаны использовать общие:

- API и модели данных;
- хранилище персонажей;
- расчёты характеристик;
- игровой движок;
- правила применения действий, эффектов и предметов;
- журнал событий;
- справочники и сущности библиотеки.

Дублировать доменную и игровую логику в мобильных компонентах запрещено.

## 2. Приоритеты и границы

### Приоритет A

- список персонажей;
- мобильный лист персонажа;
- создание персонажа первого уровня;
- повышение уровня и редактирование через мобильный мастер;
- добавление предметов, действий, заклинаний, черт, эффектов, состояний и ресурсов.

### Приоритет B

- мобильные редакторы предметов;
- мобильные редакторы заклинаний;
- мобильные редакторы черт;
- мобильные редакторы эффектов.

### Вне текущей итерации

- мастерские инструменты;
- обязательная мобильная поддержка боёв, инициативы, групп и магазинов;
- полноценная замена всех десктопных экранов;
- авторизация и мобильный сценарий входа;
- офлайн-режим и отложенная синхронизация;
- создание пользовательских сущностей непосредственно из листа;
- выбор цели как обязательная часть применения действия;
- отмена уже выполненного действия;
- автоматический переход на мобильную версию без согласия пользователя.

## 3. Маршруты

Основной префикс мобильной версии — `/m`.

| Маршрут | Назначение |
|---|---|
| `/m` | Переход на `/m/characters` |
| `/m/characters` | Список персонажей |
| `/m/characters/new` | Создание персонажа |
| `/m/characters/:id` | Мобильный лист персонажа |
| `/m/characters/:id/edit` | Редактирование персонажа |
| `/m/characters/:id/level-up` | Повышение уровня |
| `/m/characters/:id/add` | Выбор типа добавляемой сущности |
| `/m/characters/:id/add/:type` | Каталог сущностей выбранного типа |

Существующие маршруты `/characters-v3/*`, `/characters-forge` и `/character-forge/*` продолжают работать без изменения.

Между мобильной и десктопной версиями не добавляются постоянные взаимные ссылки.

## 4. Предложение мобильной версии

При открытии существующего интерфейса на телефоне или планшете пользователь получает ненавязчивое предложение открыть мобильную версию.

Требования:

- предложение не выполняет автоматический redirect;
- основное действие — «Открыть мобильную версию»;
- вторичное действие — «Не сейчас»;
- отказ сохраняется локально, чтобы предложение не появлялось при каждом переходе;
- пользователь может продолжить работу в текущем интерфейсе;
- авторизация на этом этапе не добавляется.

## 5. Общая мобильная оболочка

### 5.1. Навигация

Стартовый экран мобильной версии — список персонажей.

Внутри открытого листа глобальная навигация приложения скрыта. Нижняя панель используется только для пяти страниц листа:

1. **Общее**
2. **Действия**
3. **Инвентарь**
4. **Пассивы**
5. **Ещё**

При каждом новом открытии персонажа активна страница «Общее». Запоминать последнюю страницу между открытиями не требуется.

Кнопка «Назад» в верхней панели листа возвращает пользователя к списку персонажей.

### 5.2. Постоянная верхняя панель

На всех пяти страницах листа постоянно видны:

- имя персонажа;
- текущие и максимальные HP;
- временные HP при их наличии;
- КЗ.

Поведение:

- нажатие на имя открывает страницу «Общее»;
- нажатие на HP открывает карточку изменения здоровья;
- нажатие на КЗ открывает расшифровку расчёта;
- верхняя панель остаётся видимой при прокрутке;
- панель не должна перекрывать содержимое или системные safe area.

Эффекты, состояния и концентрация в постоянной панели не показываются.

### 5.3. Визуальный стиль

Мобильная версия сохраняет визуальный язык текущего `/characters-v3`:

- тёмный тёплый фон;
- золотые акценты;
- бумажные и фэнтезийные мотивы;
- существующую семантику цветов редкости;
- визуальное родство карточек и панелей.

Допускается более плотная компоновка. Элементы не обязаны становиться крупными только ради мобильного вида, но основные интерактивные цели должны оставаться пригодными для касания.

Hover не должен быть необходим для чтения информации или выполнения операции.

## 6. Список персонажей

### 6.1. Карточка персонажа

Карточка содержит:

- портрет;
- имя;
- вид;
- класс;
- уровень;
- текущие и максимальные HP;
- КЗ;
- индикатор активного боя при наличии;
- основную кнопку «Открыть»;
- меню дополнительных операций `⋯`.

### 6.2. Основное действие

Кнопка «Открыть» ведёт на `/m/characters/:id`.

### 6.3. Меню `⋯`

Меню содержит:

- редактировать;
- повысить уровень;
- дублировать;
- удалить.

Удаление требует отдельного подтверждения. Переход в десктопную версию в меню не добавляется.

На экране присутствует заметная кнопка создания нового персонажа.

## 7. Структура мобильного листа

### 7.1. Страница «Общее»

Содержит:

- имя;
- портрет;
- текущие, максимальные и временные HP;
- уровень;
- скорость;
- бонус мастерства;
- КЗ;
- класс;
- вид и подвид;
- инициативу;
- характеристики;
- спасброски рядом с соответствующими характеристиками;
- навыки;
- описание персонажа;
- заметки.

Нажатие на проверяемую характеристику, спасбросок, навык или инициативу открывает карточку подтверждения броска.

### 7.2. Страница «Действия»

Содержит блоки:

1. **Ресурсы**
2. **Действия**
3. **Заклинания**

Также содержит кнопки:

- новый ход;
- короткий отдых;
- долгий отдых.

Каждая операция сначала открывает карточку подтверждения. Дополнительный диалог после нажатия основной кнопки не показывается.

### 7.3. Страница «Инвентарь»

Содержит:

- инвентарь;
- экипировку;
- вложенные контейнеры, если они поддерживаются общей моделью;
- количество предметов;
- размер персонажа;
- текущий переносимый вес;
- грузоподъёмность.

Для предметов должны поддерживаться уместные отдельные операции, например:

- использовать;
- экипировать;
- снять;
- настроить количество;
- переместить в контейнер.

Конкретный набор кнопок определяется возможностями предмета и общей игровой логикой.

### 7.4. Страница «Пассивы»

Содержит блоки:

1. **Черты и способности**
2. **Чувства**
3. **Сопротивления, иммунитеты и уязвимости**
4. **Состояния**
5. **Эффекты**

Пассивные черты, способности и эффекты не имеют кнопки «Применить». Их карточка используется только для просмотра сведений и доступных служебных операций.

Активируемые сущности могут иметь собственные кнопки операций.

### 7.5. Страница «Ещё»

Содержит:

- владения оружием;
- владения доспехами и щитами;
- языки;
- журнал игровых событий;
- настройки мобильного листа.

## 8. Карточка сущности и подтверждение операций

### 8.1. Телефон

На телефоне сущность открывается как отдельный полноэкранный экран.

Структура:

- фиксированная верхняя панель с кнопкой «Назад» и названием;
- прокручиваемое содержимое;
- фиксированная нижняя область операций;
- кнопки конкретных операций;
- кнопка отмены не обязательна, если доступна верхняя кнопка «Назад».

### 8.2. Планшет

На планшете карточка открывается по центру поверх текущей страницы:

- фон затемняется;
- карточка не занимает весь экран;
- касание вне карточки закрывает её без изменений;
- положение прокрутки основной страницы сохраняется.

Рекомендуемая граница между режимами — 768 CSS px. Она может быть скорректирована после визуального тестирования, но поведение должно зависеть от доступного пространства, а не от user agent.

### 8.3. Закрытие

Без выполнения операции карточка закрывается:

- кнопкой «Назад»;
- системным Back;
- поддерживаемым жестом возврата;
- касанием по затемнению на планшете.

Закрытие не изменяет данные.

### 8.4. Применение

Общий сценарий:

1. пользователь нажимает на сущность или операцию;
2. открывается карточка;
3. пользователь при необходимости задаёт параметры;
4. пользователь нажимает конкретную кнопку операции;
5. операция выполняется;
6. карточка закрывается;
7. лист обновляется;
8. над нижней навигацией появляется уведомление.

Дополнительный подтверждающий диалог не используется: сама карточка является подтверждением.

Для сущности с несколькими возможностями показываются отдельные кнопки, например «Использовать», «Экипировать» и «Снять». Одна неоднозначная кнопка «Применить» не используется.

Выбор цели не является обязательным. Результаты бросков сохраняются в журнале событий и не обязаны показываться в уведомлении.

### 8.5. Результат и ошибки

После успешной операции карточка закрывается. Текст уведомления согласуется с типом события:

- «Действие „X“ выполнено»;
- «Заклинание „X“ применено»;
- «Предмет „X“ использован»;
- «Эффект „X“ наложен»;
- «Ресурс „X“ изменён»;
- «Начат новый ход»;
- «Короткий отдых завершён»;
- «Долгий отдых завершён».

Результаты бросков остаются в журнале.

При ошибке:

- карточка остаётся открытой;
- введённые параметры не теряются;
- сообщение показывается рядом с областью операций;
- повторное нажатие разрешено после устранения причины;
- частично применённое состояние недопустимо.

Отмена уже успешно выполненной операции через уведомление не требуется.

## 9. Добавление сущностей

### 9.1. Точки входа

Добавление доступно:

- через общую кнопку `+` в шапке листа;
- через отдельную кнопку `+` в соответствующем блоке.

Общая кнопка открывает выбор категории. Локальная кнопка сразу открывает каталог нужного типа.

### 9.2. Категории

- предметы;
- действия;
- заклинания;
- черты;
- эффекты;
- состояния;
- ресурсы.

Классовые особенности представлены через общую модель эффектов и не образуют отдельную категорию добавления.

Игрок может добавлять только готовые сущности библиотеки. Создание новой сущности из мобильного листа не поддерживается.

### 9.3. Каталог

Каталог содержит:

- заголовок и кнопку возврата;
- поиск;
- релевантные типу фильтры;
- список или сетку результатов;
- режим множественного выбора;
- счётчик выбранного;
- основную кнопку добавления.

Из результата можно открыть карточку сущности без потери текущего поиска, фильтров, выбора и положения прокрутки.

### 9.4. Множественный выбор

Пользователь может:

- выбрать несколько разных сущностей;
- указать несколько копий одной сущности, если тип это допускает;
- отменить отдельный выбор;
- очистить весь выбор;
- добавить всё выбранное одной операцией.

Добавление пачки должно быть атомарным либо явно сообщать, какие позиции не удалось добавить. Предпочтительно предварительно валидировать всю пачку и не выполнять её при наличии блокирующей ошибки.

### 9.5. Дубликаты

| Тип | Правило |
|---|---|
| Предмет | Повторное добавление увеличивает количество |
| Заклинание | Нельзя добавить повторно |
| Действие | Нельзя добавить повторно |
| Состояние | Одна активная запись данного состояния |
| Ресурс | Одна запись ресурса; повторное добавление предлагает изменить параметры |
| Черта | Повтор допустим только при признаке `repeatable` |
| Эффект | Повтор допустим только при признаке `repeatable` |

Повторяемые эффекты хранятся как независимые экземпляры. Каждый экземпляр имеет собственные:

- источник;
- длительность;
- момент начала;
- оставшееся время или число ходов;
- состояние активности;
- идентификатор экземпляра.

## 10. Мобильный мастер персонажа

### 10.1. Общие требования

Мастер создаётся как новый мобильный интерфейс с последовательными экранами и фиксированными кнопками:

- «Назад»;
- «Далее».

На последнем шаге «Далее» заменяется на «Создать персонажа» или «Сохранить».

Текущий прогресс и ошибки шага должны быть видимы. Перейти далее нельзя, пока обязательные поля текущего шага невалидны.

### 10.2. Создание

На первом этапе поддерживается создание персонажа только первого уровня.

Шаги:

1. **Основное:** имя, портрет и базовые сведения.
2. **Вид:** вид, подвид и связанные выборы.
3. **Класс:** класс и связанные классовые выборы.
4. **Предыстория:** предыстория, черта происхождения и связанные выборы.
5. **Характеристики:** распределение значений по действующим правилам.
6. **Навыки и владения:** все ещё не разрешённые выборы навыков, инструментов, языков и владений.
7. **Заклинания:** показывается только при наличии выбора заклинаний.
8. **Проверка:** итоговое резюме и создание персонажа.

Стартовое снаряжение не выносится в отдельный шаг. Оно выбирается внутри шага источника:

- классовое — на шаге класса;
- выдаваемое предысторией — на шаге предыстории;
- выдаваемое другим источником — рядом с соответствующим выбором.

Динамически неприменимые шаги пропускаются.

### 10.3. Черновик

Черновик автоматически сохраняется локально после каждого значимого изменения.

Требования:

- восстановление после закрытия вкладки или браузера;
- восстановление после перезагрузки;
- отдельный черновик для создания и каждого редактируемого персонажа;
- предложение продолжить или удалить найденный черновик;
- удаление черновика после успешного создания или сохранения;
- версия формата черновика для безопасной миграции или сброса несовместимых данных.

### 10.4. Редактирование и повышение уровня

Редактирование и level-up используют тот же мобильный мастер.

Ограничения и требования должны совпадать с десктопным мастером:

- те же доступные и заблокированные выборы;
- те же зависимости от вида, класса, подкласса и уровня;
- те же проверки конфликтов;
- те же правила выдачи и удаления сущностей;
- те же правила сохранения текущего HP, ресурсов, экипировки и игровых данных;
- те же предупреждения о последствиях изменения ранних выборов.

Level-up должен поддерживать уровни, доступные десктопной реализации. Ограничение «только первый уровень» относится только к созданию нового персонажа.

## 11. Адаптивность

### Телефон

- ширина проверки: 360–430 CSS px;
- основной режим — портретный;
- поддерживается горизонтальная ориентация;
- лист использует одну колонку;
- сущности открываются полноэкранно;
- нижняя навигация учитывает safe area;
- горизонтальный скролл страницы отсутствует.

### Планшет

- ширина проверки: 768–1024 CSS px;
- блоки могут переходить в две колонки;
- информационная архитектура и пять страниц сохраняются;
- сущности открываются центральной карточкой с затемнением;
- нижняя навигация сохраняется;
- интерфейс не превращается автоматически в существующую десктопную версию.

### Браузеры

- актуальный Safari на iOS/iPadOS;
- актуальный Chrome на Android;
- touch-взаимодействие без зависимости от hover;
- системный Back не должен приводить к потере несохранённых данных без предупреждения.

## 12. Состояние интерфейса

При открытии и закрытии карточки должны сохраняться:

- активная страница листа;
- положение прокрутки;
- состояние раскрытых секций;
- поиск и фильтры каталога;
- выбранные в каталоге сущности.

При переключении между пятью страницами данные листа не перезагружаются без необходимости.

Во время запросов:

- повторное выполнение операции блокируется;
- кнопка показывает состояние загрузки;
- навигация не должна приводить к двойной отправке;
- успешный ответ отражается в листе без ручного обновления страницы.

## 13. Доступность и взаимодействие

- все операции доступны с клавиатуры там, где она используется на планшете;
- интерактивные элементы имеют доступные названия;
- выбранный пункт нижней панели обозначается не только цветом;
- фокус удерживается внутри открытой планшетной карточки;
- после закрытия фокус возвращается к вызвавшему элементу;
- учитывается `prefers-reduced-motion`;
- текст не должен обрезаться без возможности прочитать полное значение;
- нативный масштаб страницы не блокируется.

## 14. Критерии приёмки

### Навигация

- `/m` открывает список персонажей;
- лист всегда начинает работу со страницы «Общее»;
- нижняя панель переключает пять страниц без полной перезагрузки;
- в листе отсутствует вторая глобальная нижняя панель;
- Back возвращает на ожидаемый предыдущий экран.

### Лист

- имя, HP, временные HP и КЗ доступны на каждой странице;
- все согласованные блоки находятся на своих страницах;
- ни одна обязательная операция не зависит от hover;
- действия выполняются только из карточки подтверждения;
- успешная операция закрывает карточку и создаёт уведомление;
- ошибка сохраняет карточку и введённые параметры;
- броски записываются в журнал.

### Добавление

- доступны семь согласованных категорий;
- общая и локальные кнопки `+` работают;
- множественный выбор сохраняется при просмотре карточки;
- правила дубликатов соблюдаются;
- повторяемые эффекты получают независимые экземпляры, источники и длительности;
- готовые сущности добавляются без перехода в конструктор.

### Мастер

- можно создать валидного персонажа первого уровня;
- неприменимые шаги скрываются;
- стартовое снаряжение выбирается рядом с выдающим его источником;
- черновик переживает перезагрузку;
- редактирование и level-up используют общие с десктопом правила;
- игровые данные персонажа не теряются при level-up.

### Устройства

- нет горизонтального скролла на ширине 360 px;
- лист работоспособен в портретной и горизонтальной ориентации;
- на планшете используются две колонки там, где это улучшает компоновку;
- телефонная карточка полноэкранная;
- планшетная карточка центральная и не теряет контекст;
- нижняя панель не перекрывает последний элемент страницы;
- сценарии проходят в iOS Safari и Android Chrome.

## 15. Обязательные сквозные сценарии проверки

1. Открыть список → открыть персонажа → перейти по всем пяти страницам → вернуться к списку.
2. Изменить HP через карточку → получить уведомление → увидеть новое значение во всех местах листа.
3. Выполнить действие → закрыть карточку → проверить расход ресурса и запись в журнале.
4. Применить заклинание → проверить ячейку и журнал.
5. Экипировать и снять предмет → проверить КЗ, вес и экипировку.
6. Добавить несколько предметов и одно заклинание → проверить количество и уникальность.
7. Попытаться повторно добавить уникальное действие → получить понятную блокирующую ошибку.
8. Добавить два экземпляра повторяемого эффекта с разными источниками и длительностями.
9. Создать персонажа первого уровня, закрыв и восстановив мастер из черновика.
10. Повысить уровень существующего персонажа без потери HP, ресурсов, экипировки и журнала.
11. Повторить ключевые сценарии на телефоне и планшете.

## 16. Рекомендуемый порядок реализации

1. Маршруты `/m`, оболочка, мобильное предложение и список персонажей.
2. Верхняя и нижняя панели листа, пять пустых страниц.
3. Перенос чтения общей модели персонажа в страницы.
4. Универсальный телефонный экран/планшетная карточка сущности.
5. Выполнение действий, изменение HP, отдых и уведомления.
6. Инвентарь, пассивы, эффекты, состояния и журнал.
7. Каталог и пакетное добавление сущностей.
8. Новый мастер создания и локальные черновики.
9. Редактирование и level-up.
10. Проверка устройств, ориентаций, safe area, ошибок и сквозных сценариев.
