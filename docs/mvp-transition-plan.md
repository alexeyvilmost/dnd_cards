Версия 1.0. Аудит текущего состояния → архитектурные решения → фазы → мелкие шаги
для делегирования легковесной модели. Родительские документы:
`unified-mechanics-schema.md`, `character-creator-plan.md`, `character-creator-handoff.md`.

### Контент в прод-БД (факт)
- Классы: **2** (Воин, Волшебник), только L1 (`level_progression["1"]`), `resources` классов не заданы.
- Виды: **12**, поле `lineages` пустое у всех — подвиды живут внутри choice `subfeature` механик.
- Предыстории: **16** (владения местами русскими строками — частично нормализовано `skillNormalize`).
- Черты: **77**, категория origin: 12 (есть дубли по названиям).
- Заклинания: **163**, из них 0–1 уровня **99**; `mechanics` = **0 из 163**.
- Действия: 12 (9 с механикой). Эффекты: 62 (46 с механикой).

### Чего нет (дыры до MVP)
1. **Runtime-интерпретатор** unified mechanics — исполнение действий (бросок атаки → урон → лог), трата/восстановление ресурсов, активные эффекты с временными модификаторами, ход/отдыхи. Существующий `effect_interpreter.go` — **легаси для v2/Script**, к новой системе не относится.
2. **Runtime-состояние на v3-персонаже**: нет полей ресурсов (current/max), активных эффектов, экипировки, инвентаря, лога.
3. **Экипировка/инвентарь для v3**: слоты, вес, надеть/снять, КЗ от брони (сейчас КЗ = 10+ЛВК всегда), оружие → параметры атаки (`dice:"weapon"`, `ability:"auto"` — плейсхолдеры без резолвера).
4. **Roll-движок и лог**: генерация бросков, разбивка «к20=13 + 3 (ЛВК) + 2 (БМ) − помеха…», персист лога.
5. **Механики заклинаний** 0–1 уровня (99 шт.).
6. **Контент**: 10 классов, вторые уровни, lineages, предметы предысторий, ресурсы классов.
7. Смена хода / короткий-длинный отдых для v3.

## 2. Архитектурные решения (зафиксировать до старта)

### 2.1 Где живёт интерпретатор — **TypeScript, чистый модуль `frontend/src/engine/`**
Обоснование: весь новый движок (rules-резолвер) уже на фронте; MVP —
одиночный лист без синхронного мультиплеера; требование «новые эффекты без
изменений бэкенда» выполняется данными-ориентированным интерпретатором в любом
рантайме. Правила изоляции, чтобы не отрезать путь на сервер:
- `engine/` **не импортирует React** и ничего из `pages/components`. Вход: `(character runtime state, mechanics JSON, context) → (new state, RollLog[])`.
- Вся случайность — через инъектируемый `rng` (тестируемость, воспроизводимость, будущий серверный режим).
- Бэкенд хранит `runtime_state` (jsonb) и **append-only** `event_log` — тонкие эндпоинты без логики.

⚠️ Осознанный компромисс: сервер не авторитетен. Для группового боевого режима
интерпретатор придётся переносить/дублировать на сервер (или гонять через общий
протокол событий). Формат `RollLog`/`EngineEvent` проектируем сразу как протокол.

### 2.2 Единый парсер формул — один модуль `engine/formula.ts`
Сейчас формулы (`1 + str`, `1d10 + self_level`, `character_speed`, `weapon`,
`8+prof+con`) парсятся ситуативно. До любого контента-наполнения — единый парсер
с контекстом `{abilities, pb, level, class_levels, weapon, speed}` и словарём
переменных из схемы §8. Все места (интерпретатор, превью карточек, лист) ходят
только через него.

### 2.3 Идентификаторы — слаги, не UUID, в механиках
В `grant_spell`/`grant_feat`/choice-item используются слаги. Зафиксировать:
**в механиках всегда `card_number`-слаг** (`SPELL-xxx`/явный slug), резолв в
сущность — через реестр (см. риск R3). Добавить `slug` заклинаниям при
наполнении механик (или использовать существующий `card_number`).

### 2.4 Экипировка — на v3, старую систему не трогаем
Новые поля на `characters_v3` (`equipment` jsonb: slot→card_id; `inventory`
jsonb или отдельная таблица `character_v3_items`). Инвентарь v1/v2 замораживаем.

## 3. Риски и блокеры (что уже сейчас может помешать)

- **R1. Два параллельных мира.** Инвентарь/экипировка/боевые эндпоинты живут в v2 (`UseAction`, `CalculateArmorClass`, `EquipItem`), а новый персонаж — v3. Если начать «дотягивать» v2 — получим двойную поддержку навсегда. Решение: §2.4, v2 не трогаем вообще.
- **R2. Смешение id-пространств заклинаний.** `CharacterForge` уже мержит `draft.spellIds` (UUID) с `ruleState.spells.known` (слаги из `grant_spell`) в один массив — это сломает лист, как только у вида/класса появится грант заклинания. Чинить в первую очередь (шаг A1).
- **R3. Нет реестра слаг→сущность.** `grant_spell value:"light"` не найдёт заклинание «Свет» без маппинга. Нужен резолвер slug→entity (по `card_number`, с отчётом о битых ссылках).
- **R4. Валидации механик нет.** Конструктор позволяет raw-JSON; интерпретатор будет падать на мусоре. Есть `mechanics.schema.json` — включить валидацию при сохранении + скрипт-линтер всего контента в БД.
- **R5. `rule_state` — снапшот с фронта.** При изменении формул старые персонажи хранят устаревшие снапшоты. Правило: снапшот — только кэш, лист всегда пересчитывает (уже так); версионировать `version: 1` и мигрировать лениво.
- **R6. Формулы-плейсхолдеры без спецификации** (`dice:"weapon"`, `ability:"auto"`, `1 + str`). Пока нет `engine/formula.ts`, каждый новый потребитель добавляет свой парсинг (уже 3 места: превью, rules, старый интерпретатор). Замораживает контент-конвейер — сначала A2.
- **R7. Дубли и грязь контента.** 77 черт с дублями имён, владения предысторий русскими строками, `origin_feat` по названию, пустые `lineages`. Нужен аудит-скрипт и чистка до массового наполнения.
- **R8. Деплой фронта на Railway не срабатывает автоматически** — каждый релиз требует ручного redeploy; проверки идут через локальный `vite preview`. Операционный риск: прод отстаёт от main. Разобраться с сервисом (или добавить fallback-инструкцию).
- **R9. Нет тестов движка.** Для делегирования лёгкой модели обязательны golden-тесты интерпретатора (фикстуры «механика+состояние → события+состояние»), иначе регрессии не поймать. Vitest в репо не настроен.
- **R10. Ресурсы классов не заданы** (`resources: None`) — Второе дыхание тратит `second_wind`, которого у персонажа нет. Схема ресурсов класса → max-пул персонажа должна появиться в движке (D2) и в контенте классов.

## 4. Высокоуровневый план (фазы)

```
A. Фундамент: гигиена id, формулы, валидация, тесты        (разблокирует всё)
B. Roll-движок + лог событий                                (ядро прозрачности)
C. Экипировка и инвентарь на v3 + КЗ/оружие                 (предметы функциональны)
D. Runtime: ресурсы, активные эффекты, ход, отдыхи          (действия функциональны)
E. Интерпретатор действий: attack_roll/save/check/auto      (единый исполнитель)
F. Лист персонажа: интеграция движка, детализация, броски   (UX)
G. Контент-конвейер: классы×2 ур., виды, предыстории,
   черты, заклинания 0–1, предметы                          (масштаб через данные)
H. Стабилизация: сквозные прогоны, отчёт покрытия, деплой   (готовность к бою)
```

Зависимости: A → (B, C) → D → E → F; G идёт параллельно после A (инструменты)
и завершается после E (проверка функциональности). H — финал.

## 5. Шаги для легковесной модели

Формат: **Шаг (≈1 сессия)** — задача → критерий приёмки (КП).
Общие правила для исполнителя: билд `go build ./...` и `npx vite build`
(не `npm run build`); jsonb, не text[]; push в main деплоит бэкенд;
фронт проверять `npx vite preview --port 5173` против прод-бэкенда;
кириллица в curl — `--data-binary @file`; тесты движка — обязательны.

### Фаза A — фундамент
- **A0. Vitest.** Подключить vitest к frontend (без jsdom, только node), `npm run test`. КП: пустой тест проходит, добавлен в README-заметку.
- **A1. Развести id-пространства заклинаний.** В `ruleState.spells.known` хранить слаги; в листе/форже резолвить слаги через реестр (A3), UUID-выбор игрока хранить отдельно; убрать мерж UUID+слагов. КП: персонаж с `grant_spell` от вида показывает заклинание в листе; тест.
- **A2. `engine/formula.ts`.** Парсер формул схемы §8: кости `NdM`, `+−*/`, переменные (`str..cha`, `prof_bonus`, `self_level`, `class_level:<id>`, `spellcasting`, `character_speed`), спец-значения `weapon`/`auto` возвращаются как маркеры. `evaluate(formula, ctx)` и `describe(formula, ctx)` (для лога: «1к10 + 3 [ТЕЛ]»). КП: таблица тестов ≥20 формул из реального контента.
- **A3. Реестр сущностей по слагам.** `engine/registry.ts`: async-резолв `spell|action|effect|feat: slug → entity` по `card_number` с кэшем + функция отчёта битых ссылок. КП: тест + скрипт `scripts/audit-refs.mjs`, печатающий все битые слаги в прод-контенте.
- **A4. Валидация механик при сохранении.** В конструкторах (raw-JSON и блоки) прогонять `mechanics.schema.json` (ajv), показывать ошибки; линтер-скрипт по всей БД. КП: сохранение мусора блокируется; отчёт по прод-контенту чистый или с известным списком.
- **A5. Аудит и чистка контента.** Скрипт: дубли черт, ру-строки владений → id, `origin_feat` название → slug, пустые lineages. Починка данных через API. КП: повторный прогон аудита — 0 замечаний.

### Фаза B — броски и лог
- **B1. Типы протокола.** `engine/events.ts`: `RollLog` (кость, выпало, модификаторы[{value, source, reason}], advantage-состояние, итог, цель/DC), `EngineEvent` (action_used, roll, damage, heal, resource_spent, effect_applied/expired, rest, turn). КП: типы + фабрики + тесты сериализации.
- **B2. Roll-движок.** `engine/roll.ts`: d20 с адв/дизадв (обе кости в лог), крит-диапазон, суммирование модификаторов из списка с источниками, дайс-выражения из A2. Инъекция rng. КП: golden-тесты с фиксированным rng.
- **B3. Персист лога.** Бэкенд: `character_events` (id, character_id, ts, type, payload jsonb) + POST batch/GET по персонажу. КП: round-trip через curl.
- **B4. UI лога.** Панель «Журнал» в листе: события с разворотом broll-разбивки («к20: 13 → +3 ЛВК (модификатор) +2 БМ (владение) = 18 vs КЗ 15 — попадание»). Простая визуализация броска (анимация числа/кости — компонент `RollFlash`). КП: скриншот, событие из движка отображается.

### Фаза C — предметы и экипировка (v3)
- **C1. Runtime-модель на бэкенде.** Миграция: `characters_v3` + `equipment` (jsonb slot→card_id), `inventory_items` (jsonb массив {card_id, qty}), `resources` (jsonb current), `max_resources` (jsonb), `active_effects` (jsonb), `turn_state` (jsonb). PUT-совместимость (не затирать при старом клиенте — отдельный PATCH `/characters-v3/:id/runtime`). КП: round-trip.
- **C2. Инвентарь UI.** Секция «Инвентарь» в листе: добавить предмет (поиск по cards), удалить, количество, суммарный вес vs грузоподъёмность (STR×15). КП: сквозной сценарий в браузере.
- **C3. Надеть/снять.** Слоты (head/body/one_hand×2/two_hands/…), конфликт слотов (двуручное блокирует обе руки), требования предмета. КП: экипировка меча и щита у Воина; конфликт двуручного — ошибка.
- **C4. КЗ-конвейер.** `engine/ac.ts`: base (armor formula | set_value из эффектов типа Защиты без доспехов) + щит + модификаторы, с разбивкой источников. Лист показывает breakdown в tooltip. КП: тесты (без брони, кожаная, кольчуга+щит, unarmored defense).
- **C5. Оружие → параметры атаки.** Резолвер `weapon context`: из экипировки достать кость (`bonus_value`), тип урона, свойства (fin/versatile/two-handed/light); `ability:"auto"` → str/dex по свойствам; `dice:"weapon"` → кость. Базовые действия-атаки показывают реальные цифры персонажа. КП: тесты + карточка «Атака оружием» в листе с настоящей костью.
- **C6. Магазин → v3.** Покупка кладёт в инвентарь v3, списывает валюту (поле `currency` jsonb на v3). КП: сквозной сценарий покупки.
- **C7. Сид предметов.** Всё базовое оружие PHB (simple/martial), щиты, доспехи (лёгкие/средние/тяжёлые) с корректными полями. КП: аудит-скрипт покрытия предметов.

### Фаза D — runtime персонажа
- **D1. Модель ресурсов.** Схема `class.resources` → max-пулы персонажа (формулы из A2: «2», «prof_bonus», по уровню), инициализация при создании/левелапе, отображение чипов ресурсов в листе. КП: у Воина L1 second_wind 2/2 (по классу), у перса — heroic_inspiration от Человека.
- **D2. Трата/проверка стоимости.** `engine/cost.ts`: проверить и списать `activation.cost[]` (мультиресурс), `uses` (count/per). Действие недоступно (серое) без ресурса. КП: тесты; Второе дыхание тратится и сереет.
- **D3. Ход и отдыхи.** Кнопки «Новый ход» (восстановить action/bonus/reaction, тикнуть длительности активных эффектов), «Короткий отдых», «Длинный отдых» (правила восстановления per из uses/ресурсов). События в лог. КП: сценарий: потратил действие → новый ход → доступно.
- **D4. Активные эффекты.** Применение `Payload.modifier/resistance/condition` как активного эффекта с длительностью; их учёт в бросках (через B2 источники) и отображение в листе (иконки с таймером). Снятие по длительности/вручную. КП: «Уклонение» вешает эффект, атаки по себе — с помехой в логе, эффект уходит в начале следующего хода.

### Фаза E — интерпретатор действий
- **E1. Каркас исполнителя.** `engine/execute.ts`: `executeAction(state, mechanics, ctx) → {state', events[]}`; маршрутизация по `resolution`; учёт `targeting` (цель обязательна/self). КП: golden-тесты на 4 категории-заглушки.
- **E2. `attack_roll`.** Бросок (B2) + модификаторы из активных эффектов/пассивок (`applies_to.roll:"attack"`), `vs ac` цели (пока цель = вручную введённое КЗ или выбранный персонаж), `on_hit/on_crit` → урон через формулы + weapon context (C5). КП: все три базовые атаки исполняются с логом; эффект «преимущество на следующую атаку» влияет на любую из них (ключевой тест унификации!).
- **E3. `save` и `ability_check`.** DC из формулы, `who:target/self`, `on_fail/on_success`, contest (Толкнуть). Цель-персонаж или ручной ввод модификатора цели. КП: Толкнуть и Оружие дыхания исполняются с логом.
- **E4. `auto` + payload-исполнители.** healing (Второе дыхание), temp_hp, resource grant, movement (лог-запись), condition apply, narrative (просто в лог). КП: Второе дыхание лечит с разбивкой формулы.
- **E5. Заклинания через тот же исполнитель.** Спелл = mechanics + cost слота (слоты как ресурс `spell_slot_1`); заговоры бесплатные. Скейлинг заговора по уровню персонажа (`scaling.per`). КП: Огненная стрела (атака), Огненные ладони (спас), Лечение ран (авто) исполняются.

### Фаза F — лист персонажа (интеграция)
- **F1. Панель действий.** Секция «Действия» с кнопками: базовые + от класса/вида + подготовленные заклинания; серость по стоимости; клик → (опц. выбор цели/DC) → исполнение → лог. КП: сквозной ход Воина: атака, Второе дыхание, конец хода.
- **F2. Детализация значений везде.** Общий компонент `ValueBreakdown` (tooltip/popover): КЗ, HP max, инициатива, скорость, спасброски, атака/урон карточек. Данные — из engine (источники). КП: у каждого числа листа есть разбивка.
- **F3. HP-виджет.** Урон/лечение вручную + от событий движка, temp HP, состояние при 0 (без death saves в MVP — просто «без сознания»). КП: сценарий урона/лечения с логом.

### Фаза G — контент (параллелится после A, батчами)
Каждый шаг: сид через API + аудит-скрипт + ручная проверка одного персонажа.
- **G1. Инструментарий контента.** Общий сидер-каркас (идемпотентный, слаги), отчёт покрытия: по классу/виду/предыстории — чего не хватает (ресурсы, механики, действия). КП: `scripts/coverage-report.mjs`.
- **G2. Классы 1–6 (Воин, Волшебник, Плут, Жрец) × уровни 1–2.** Умения как эффекты/действия с механиками, `level_progression`, `resources`, `skill_choices`, спелл-листы кастеров. КП: отчёт покрытия зелёный; персонаж каждого класса собирается и все действия исполняются.
- **G3. Классы 5–8 (Варвар, Монах, Паладин, Следопыт) × 1–2.** КП: аналогично.
- **G4. Классы 9–12 (Бард, Друид, Чародей, Колдун) × 1–2.** КП: аналогично.
- **G5. Виды: заполнить `lineages` структурно** + доверить все 12 видов PHB-2024 (эффекты уже есть у 10) + действия видов (Оружие дыхания как действие и т.п.). КП: выбор любого вида+подвида даёт рабочие способности.
- **G6. Предыстории 16: предметы.** `equipment_options` A/B → реальные card_id, выдача в инвентарь при создании; владения → id; черта → slug. КП: создание персонажа кладёт стартовый инвентарь.
- **G7. Черты происхождения 10: механики.** Alert/Skilled/Tough/Savage Attacker/Lucky/Magic Initiate×3/Musician/Crafter, дубли почищены (A5). КП: каждая черта даёт исполняемый эффект/ресурс.
- **G8. Заговоры (~30): механики** батчами по 10 (attack_roll/save/auto; нарративные — `narrative`). КП: линтер схемы зелёный; 5 выборочных исполняются.
- **G9. Заклинания 1 уровня (~69): механики** батчами по ~15. КП: аналогично.

### Фаза H — стабилизация
- **H1. Сквозные сценарии.** 4 эталонных персонажа (Воин/Человек, Волшебник/Эльф, Плут/Полурослик, Жрец/Дварф): создание → лист → экипировка → 3 хода с действиями → отдых. Чек-лист в docs. КП: все проходят в браузере на проде.
- **H2. Отчёт готовности.** coverage-report по всем требованиям MVP (таблица из этого документа) + известные ограничения. КП: документ в docs.
- **H3. Деплой.** Разобраться с авто-деплоем фронта на Railway (или задокументировать ручной процесс). КП: push в main обновляет прод-фронт (или инструкция).

`A0→A1→A2→A3→A4→A5` → `B1→B2→B3→B4` → `C1→…→C7` → `D1→…→D4` →

если исполнителей двое.
