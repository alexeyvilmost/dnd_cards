# Роадмап движка — следующие шаги (2026-07-09)

Составлен по веерной разведке кода+планов (6 читателей → синтез → адверсариальный критик).
Приоритеты уже включают 4 поправки критика. Доказательства — как `файл:строка`.

## Диагноз: где мы на самом деле

**Фундамент A–F РЕАЛЕН, не заглушки** (подтверждено всеми шестью отчётами по коду):
шина событий с подписками и реакциями end-to-end (`dispatch.ts:62-85`, `execute.ts:791-822`,
реакция исполняется через `executeAction` из `SheetHpPanel.tsx:159`), formula-aware конвейер
модификаторов на общем пути лист+бой (`modifiers.ts`, `breakdown.ts:60-64`), двусторонний контекст
(`who:'target'`, `projectedAgainst`, спас берёт моды цели), состояния-данными, настоящий стекинг
(`execute.ts:250-274`), конвейер входящего урона сопротивление→концентрация→реакции (C15,
`execute.ts:879-941`). **Переделывать фундамент не надо.**

**Настоящая граница — «ПРИМЕНЕНИЕ ПЕЙЛОАДОВ». Два независимых чокпоинта, на которых застревает
data-driven контент:**
1. **Пассивы:** `collectPassiveMechanics` (`resourceInit.ts:11`) отдаёт `mechanics` эффекта КАК ЕСТЬ,
   не разворачивая `choice` → runtime-пейлоады из выбора (resistance/modifier/condition/temp_hp) не
   доходят ни до `resistanceLevelFor`, ни до `collectModifiers`. (Граница из контент-свипа.)
2. **Рантайм:** роутер `applyPayloads` (`execute.ts:534-566`) молча роняет `resistance / set_value /
   grant_action / variable` в `NOT_IMPLEMENTED` — **данные/схема эти kind знают** (`mechanicsView.ts:9-10`),
   **интерпретатор игнорирует**. Это прямое нарушение «без хардкода» наоборот: данные есть, кода нет.

Обе — дешёвые S/M-правки, оживляют максимум контента на единицу усилий, лежат ровно в парадигме
«без хардкода». Крупные пункты (полноценный 2D-бой) упираются в **отсутствующий EncounterState** —
см. ярус 4, это отдельное фундаментальное решение, а не «пункт на M».

## Ярус 0 — дешёвая корректность (S). ✅ ВЫПОЛНЕНО
Все пять правок сделаны + прошли адверсариальное ревью (3 лизны). Тесты: `engine/yarus0.correctness.test.ts`
(15 дискриминирующих) + правка `circumstances.test.ts`. Гейты: tsc 0, 307 vitest, 83 mvp.

**Правки ревью (сверх исходных пяти):**
- Состязание проверок теперь кидает ОДИН бросок по выгоднейшему навыку цели (RAW 2024), а не максимум
  из нескольких d20 (это давало защите фактическое преимущество). Толчок/Подножка проходят по правилам.
- Вторая половина 0.5: нераспознанный предикат-гейт → **false** (closed-by-default), а не «истина по
  умолчанию» — модификатор-ограничитель больше не висит всегда. narrative — намеренное исключение (→ true).
- `applyResistancePayload` уважает `duration.rounds` (тик хода снимает), иначе `manual` — как чип Ярости.

**Отложено (задокументировано, не регресс):**
- prone не проецирует преимущество ближних атак (нужны дальность/позиции → EncounterState, ярус 4).
- Полная жизнь «стоячих» резистов/модификаторов (снятие по источнику при конце Ярости) — C6 (ярус 2.2).
- `runAbilityCheck` не форсит планирующую ветку (кости в on_success состязания) — редко, риск dice-plan; после C6.
- «Голая» проверка характеристики (ability без skill) потеряла безусловный БМ — верно по RAW; нужен контент-свип перед выкатом.

Видимые баги того же класса, что уже закрытый C14: считают НЕВЕРНО для реальных персонажей уже

## Ярус 1 — контент-рычаг: «применение пейлоадов» (M)
Два независимых M-чокпоинта. **Честно:** по немедленному охвату `by-trigger` (~20) может превосходить
`choice-разворот` (~7); первенство разворота — в том, что это ФУНДАМЕНТ под весь будущий data-driven
контент с выбором, а не в сиюминутном охвате.

**1.1 ✅ СДЕЛАНО.** Общие helpers ключа/выбора вынесены в `mechanics/expandChoices.ts` (резолвер
импортирует их — ключ провабельно идентичен: `passiveSourceId` == `sourceFromOrigin(...).id` == `choiceKey`
форге, подтверждено ревью). `collectPassiveMechanics(assembled, resolvedChoices)` эмитит выбранные
рантайм-пейлоады синтетической `auto`-механикой (её уже читает `payloadsOf`). Набор пассивных kind
СУЖЕН до реально потребляемых `{modifier, resistance, set_value}` (resource/variable/condition убраны —
нет потребителя на пути пассивов). Правка ревью: apply-шаблон `damage_type` (`{kind:'resistance',
value:тип}`) нормализуется в рабочую форму (`damage_type`+уровень). Тесты `mechanics/expandChoices.test.ts`
(13, вкл. сквозной choice→resistanceLevelFor→урон вдвое, modifier→collectModifiers, вложенный choice,
рассинхрон ключа). Гейты tsc 0 / 320 vitest / 83 mvp. **ЧТОБЫ ОЖИВИТЬ ~7 эффектов** — их надо
конвертировать narrative→choice (контент-задача группы A свипа); движок теперь готов их применить.

- **1.1 (исходный план) Разворот выбранной `choice`-ветки в `collectPassiveMechanics`** (M). Оживляет ~7 эффектов
  сопротивления из выбора сразу (аасимар/тифлинг, EFFECT-0235/0139/0160/0092) + задел под ~37
  отложенных группы A. Вынести `selectedChoicePayloads/choiceInstanceId/sourceFromOrigin` из
  `resolveCharacterRules.ts:71-97,35-45` в новый `mechanics/expandChoices.ts` и звать ВЕРБАТИМ (иначе
  рассинхрон ключа `choiceInstanceId` → тихий no-op); сигнатура `collectPassiveMechanics(assembled,
  resolvedChoices)`, протянуть `resolvedChoices` в 4 колл-сайта (`CharacterSheetMVP.tsx:207`,
  `SheetActionsPanel.tsx:159`, `SheetRestButtons.tsx:37`, `resourceInit.ts:41`); каждый `{kind:'choice'}`
  → `{resolution:'auto',result:[...]}`, фильтр `kind∈PAYLOAD_KINDS`, `grant_*` не тащить.
**1.2 ✅ СДЕЛАНО.** Выбор context:'in_play' ВНУТРИ действия разрешается на клике: `runViaDialog`
(SheetActionsPanel) делает предпроход `collectInPlayActionChoices` → новый promise-`ChoiceDialogProvider`
(зеркало DiceDialog, рендерит `ChoiceResolver`) → строит `ctx.choices` (плоская карта на одно исполнение,
ключ = СЫРОЙ choice.id) → `execCtx(rng, planning, choices)`. Движок: `applyPayloads` получил `case 'choice'`
(читает `ctx.choices[p.id]`, разворачивает выбранную ветку тем же роутером, рекурсия для вложенных),
`runMechanicEffects` обрабатывает top-level `{kind:'choice'}`. Отмена выбора = отмена действия (до списания
стоимости — подтверждено ревью). Правки по 3-лизному ревью: (1) fallback id согласован коллектор↔движок
(`?? 'choice'`) — иначе безымянный choice был тихим no-op; (2) count>1 поддержан (`ctx.choices` = `string|string[]`,
применяются ВСЕ выбранные ветки); (3) resistance из apply-шаблона нормализуется и в рантайме
(`normalizeChoicePayload`); (4) build-гранты (`grant_*`) из выбора фильтруются на рантайм-пути (нет мусора
NOT_IMPLEMENTED); (5) гард гонки resolver в диалоге. Тесты `engine/inplayChoice.test.ts` (13). Гейты tsc 0 /
333 vitest / 83 mvp; лист рендерится. **ЧТОБЫ ОЖИВИТЬ ~20 эффектов группы B** — смоделировать их выбор как
`choice(context:'in_play')` в действии (контент). **Отложено (задокументировано):** вложенный choice→choice в
момент действия (нужен многошаговый диалог с прогрессивным раскрытием — предпроход собирает 1-й уровень,
движок 2-й тихо пропускает без краша); коллизия двух безымянных choice в одной механике (контент даёт уникальные
id); триггерные/реакционные выборы (при Ярости шиной, а не на клике) — отдельный поздний подслайс.

- **1.2 (исходный план) By-trigger: предпроход выбора на клике действия** (M, группа B ~20 эффектов). Каркас уже есть:
  `ExecuteContext.choices` объявлен (`contracts.ts:152`), двухпроходный паттерн план→вопрос→прогон
  готов на костях `diceDialog`. Не хватает такого же предпрохода для выборов: `collectInPlayActionChoices(mech)`
  перед реальным прогоном в `SheetActionsPanel.tsx:282-294` (переиспользовать `ChoiceResolver`), читать
  `ctx.choices[key]` в новых ветках `kind:'choice'` (`execute.ts:532-568,733-749`). Триггерные/реакционные
  выборы (при Ярости, реакцией) — отдельный поздний подслайс.

## Ярус 2 — фундаментальные долги ядра (M). Разблокируют ремеханизацию контента (ярус 3)
- **2.1 C4 — очередь событий вместо прямой рекурсии `emitEvent`** (M). **Живой краш-риск уже сейчас:**
  `emitEvent→runMechanicEffects→runAttackRoll→emitEvent` без лимита; при `PLANNING_RNG=0.94` on-hit-райдер
  зацикливается детерминированно, а Скрытая атака УЖЕ авто-триггерит on-hit. FIFO-очередь + лимит каскада
  ~8 событий/действие + правило «слушатель не реагирует на своё событие» + `fired.add` ДО запуска. `execute.ts:791`.
- **2.2 C6 — единый резолвер длительностей пейлоадов** (M). `applyModifierPayload` не ставит `roundsLeft`
  из `duration.rounds` (в отличие от `applyCondition`) → бафф «+2 на 3 раунда» висит вечно. `resolveDuration()`
  для ВСЕХ payload-kind + тик в start/endTurn (`turn.ts:144` уже есть) + longRest обнуляет `hp.temp`.
- **2.3 C5 — алгебра модификаторов: `mode` + `priority`** (M, Foundry-модель). `collectFromPayload` понимает
  только advantage/disadvantage/add; `op set/multiply/reroll` молча отбрасываются, нет Upgrade/Downgrade →
  «Пояс силы огра» (флагман парадигмы №3) невыразим данными. `modifiers.ts`.
- **2.4 `set_value` / `variable` в роутере** (M, БЕЗ `grant_action` — см. ниже). `set_value` (диспетч по
  target hp/ac_base/переменная), `variable` (мутация). Закрывает разрыв нормализатор↔интерпретатор для
  Неумолимой стойкости (`set_value hp=1`). `execute.ts:534-566`.

## Ярус 4 — 2D-бой. ЗАБЛОКИРОВАН отсутствующим EncounterState — стратегическая развилка
Критик подтвердил: **`EncounterState` существует только в комментариях** (`execute.ts:785`, `dispatch.ts:25-27`),
`trigger.subject` намеренно не фильтруется — мир пока ОДНОАКТОРНЫЙ, боевой 2D-поверхности нет вовсе. От него
зависит ЦЕЛАЯ модальность, ни один шаг которой сейчас нельзя ни сделать, ни проверить:
- **Исходящий урон `who:'target'`** не режется сопротивлением цели и не списывает её HP внутри движка —
  ветка `damage` (`execute.ts:535-542`) только пушит `damageEvent`; сопротивление живёт лишь в
  `applyIncomingDamage` (`execute.ts:892-897`). Прямой блокер «атака игрока по врагу».
- **Защитная сторона событий:** `PLANNED_EVENTS` (attack_roll_made, creature_leaves_reach, spell_cast врага)
  не эмитятся → Щит на входящую атаку, Контрзаклинание, Атака при возможности как реакция-защита невыразимы;
  нужно ретро-окно броска (пауза `before_hit` с мутабельным результатом).
- **C16 (зоны/эманации/summon)** — секция targeting ни на что не влияет, `summon`→NOT_IMPLEMENTED; AoE невыразимы.
- **Модальность КАМПАНИИ** (парадигма создание→лист→2D→кампания) не адресована вообще.

> **Решение развилки — за владельцем:** либо (а) явно ПРИПАРКОВАТЬ всю 2D-модальность до отдельного
> проектирования `EncounterState`+многоактор+боевая поверхность; либо (б) сделать `EncounterState`
> ближайшим фундаментальным шагом-предшественником. Оценивать его как «L-пункт» нельзя — это новая опора.
