# Фундамент правил персонажа

## Назначение

Этот слой не является боевым движком. Он отвечает за постоянную резолюцию состояния конкретного персонажа: создание, редактирование, лист, повышение уровня, экипировка, временные эффекты, результаты действий и будущий бой должны обращаться к одной и той же модели.

## Фундамент D&D 2024

В базовой модели персонажа считаем неотъемлемыми:

- 6 характеристик: `str`, `dex`, `con`, `int`, `wis`, `cha`.
- Модификатор характеристики: `floor((score - 10) / 2)`.
- Бонус мастерства по уровню: 1-4 `+2`, 5-8 `+3`, 9-12 `+4`, 13-16 `+5`, 17-20 `+6`.
- 18 навыков и их обычные характеристики.
- Владения навыками, спасбросками, инструментами, языками, оружием и доспехами как множества без стекания дублей.
- Экспертиза как отдельный слой поверх владения.
- Derived snapshot: бонусы навыков, спасбросков, КД, HP, скорость, инициатива, пассивное Восприятие.

## Расширяемые источники

Любой источник, который может изменить персонажа, должен приводиться к `RuleSource`:

- `species`, `class`, `background`, `feat` — постоянные источники создания и повышения уровня.
- `item` — экипированные предметы.
- `temporary_effect` — временные эффекты от действий, заклинаний и способностей.
- `condition` — состояния.
- `action_result` — результат использования действия.

MVP уже использует постоянные источники и резервирует форму для runtime-источников.

## Resolver

Главная функция слоя правил: `RuleInput -> CharacterRuleState`.

Она должна быть чистой и переносимой: без React, роутера, API-запросов и глобального состояния. UI и backend готовят входные данные, resolver возвращает сериализуемый snapshot.

`CharacterRuleState` хранит:

- итоговые proficiencies/expertise;
- derived values;
- `appliedGrants` с источниками каждого результата;
- `conflicts`, например повторное получение одного и того же навыка.

## Правило дублей

Владение одним навыком может быть получено только один раз. Если второй источник пытается дать тот же навык, resolver возвращает конфликт `duplicate_proficiency`; UI должен запрещать такой выбор заранее, но лист персонажа тоже обязан показывать конфликт, если он уже есть в сохранённом состоянии или возник из runtime-источника.
