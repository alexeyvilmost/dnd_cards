# Архитектурный обзор: движок эффектов / триггеров / событий / разрешений (D&D 2024)

> Дата: 2026-07-07. Автор ревью: глубокое исследование (сеть + официальные книги
> `officials/` + чтение кода + состязательная проверка утверждений).
> Статус: аналитический документ. Все «негативные» утверждения («X не реализовано»)
> проверены отдельными агентами-скептиками против кода — см. §11.

Область: сущности `effect`/`action`/`spell`, единое поле `mechanics`, интерпретатор
`frontend/src/engine/*`, система переменных, ресурсы, ход/отдых, состояния, превью.
Ориентир качества — Baldur's Gate 3. Сверка с 4 парадигмами проекта в §6.

## 0. Резюме (TL;DR)

**Вердикт.** Фундамент правильный и на удивление зрелый: единая исполняемая
`mechanics`-схема, один интерпретатор, 4 категории разрешения, полностью
data-driven система переменных с мягкой деградацией. Это уже соответствует
парадигме №1 на оси «эффект → действие → заклинание». Но три из «сквозных» обещаний
проекта пока держатся на честном слове:

1. **Нет шины событий / диспетчера триггеров.** Весь словарь `event`/`timing`/
   `subject`/`circumstances` авторится в данных, но исполняется только `long_rest`.
   Реакции (Щит, Адское возмездие, Атака при возможности, Часовой) и он-хит-райдеры
   (Скрытая атака, Божественная кара, Внезапный удар, авто-проверка концентрации)
   **не срабатывают сами**. Движок — pull-based пересчёт, без push. Это **центральный
   пробел**: без него «движок как универсальный субстрат для 2D-боя» невозможен.
2. **Парадигма №3 (методы расчёта) не реализована по-настоящему.** КЗ — это
   захардкоженный if/else-каскад, а не «набор кандидатов, берём максимум применимого».
3. **Обстоятельства (`when`/`circumstances`) не вычисляются вообще** — целая ветка
   §5.5 схемы авторится, но рантайм её игнорирует.

**Топ-6 фундаментальных изменений** (детали в §7):

| # | Изменение | Закрывает |
|---|---|---|
| A | Шина событий + диспетчер триггеров/реакций (модель Interrupt из BG3) | пробел №1, реакции, он-хит, авто-концентрация, ОА |
| B | Настоящие «методы расчёта» — реестр кандидатов, `max` применимого | парадигма №3, КЗ, СЛ, характеристики, скорость |
| C | Единый formula-aware конвейер модификаторов + вычисление `when` | два расходящихся пути, пробел №2 |
| D | Состояния и стекинг как данные (сущность Condition, StackId, «одноимённое не складывается») | нарушение парадигмы №1, RAW-стекинг 2024 |
| E | Двусторонний контекст боя (attacker + target); применение сопротивлений/урона | односторонность движка, 2D-бой |
| F | Общий `describe(mechanics) → текст`, единый для конструктора/листа/боя | парадигма №2 «превью везде» |

**Кардинально (§8):** событийно-источниковый (event-sourced) `RuleState`-редьюсер —
персонаж как проекция потока событий; и вынос движка в общий TS-пакет, исполняемый и на
клиенте, и в серверной автоматизации боя (n8n), и в кампании. Это буквальная реализация
«единого интерпретатора на всём пути».

## 2. Карта системы «как построено» (as-built)

**Сущность.** `effect`/`action`/`spell` = «презентация» (name/description/icon/
card_number) + исполняемое поле `mechanics` (jsonb). Есть легаси-поле `script` (старые
владения/сопротивления) — сосуществует. `effect_type` (10 значений) — только для
фильтрации/группировки UI; **движок поведение по нему не разветвляет** (хорошо).

**Форма `mechanics`** (§3 схемы):
```
{ activation:{mode,cost[],trigger?,requirements?}, targeting?, effects:[Interaction], duration?, uses? }
```
`Interaction.resolution ∈ {attack_roll, save, ability_check, auto}` с исходами
`on_hit/on_crit/on_miss/on_fail/on_success/result`, каждый — список `Payload`.

**Интерпретатор** (`execute.ts` `executeAction`): списывает `cost` (`canPay`/`pay`);
для каждого `effect` роутит по `resolution`; `applyPayloads` — единый роутер по
`kind` (damage/healing/temp_hp/condition/resource/modifier/movement/boon/reroll/
transform/narrative). Мягкая деградация: `MissingVariableError`/`FormulaError` →
эффект пропускается с логом, действие не падает (0 крашей на прод-контенте).

**Модификаторы броска** (`modifiers.ts` `collectRollModifiers`): pull-based — при
броске сканирует `activeEffects` + `passives`, собирает `modifier`-payload'ы под нужный
`applies_to.roll`/`filter`. `combineAdvantage`: adv+adv=adv, adv+dis=none.

**Переменные** (`character/variables.ts`): `Variable = {variable_id, var_type:number|dice,
default_value}`; значение задают эффекты `{kind:variable, op:set|add|remove}`; уровневый
рост — отдельные эффекты по уровням; мультикласс сворачивается нативно; токен формулы.

**Ресурсы** (`engine/resources.ts`, `cost.ts`): именованные пулы (`ki/focus`, `rage`,
`bardic`, `superiority`, `spell_slot_N`) из class-таблиц + грантов; recharge-метаданные;
`shortRest`/`longRest` восстанавливают.

**Ход/отдых** (`turn.ts`): `startTurn` восстанавливает action/bonus/reaction и тикает
`roundsLeft`/`start_of_next_turn`; `longRest` чистит `activeEffects` и пополняет всё;
триггер-режим исполняется **только** для `long_rest`.

**Состояния** (`conditions.ts`): захардкоженная TS-карта `CONDITION_RULES` (13 состояний
2024), маппящая состояние в модификаторы **собственных** бросков носителя; асимметрия
(«по вам с преимуществом», авто-крит, авто-провал СИЛ/ЛВК) — только текст `note`.

**КЗ** (`ac.ts` `computeAC`): if/else-каскад armor → `set_value ac_base` (первый!) → 10+dex.

**Превью** (`ActionPreview.tsx`): рендерит **свободный текст** `description`;
`summarizeMechanics` (`blocks.ts`) строит сводку из состояния блочного редактора, не из
произвольного сохранённого `mechanics`.

## 3. Сильные стороны (проверено)

1. **Единый исполняемый формат + один интерпретатор.** `effect`/`action`/`spell` —
   частные случаи `MechanicCard`; ветвление только по `activation`. Один `applyPayloads`
   на все исходы. Это ядро парадигмы №1 и «универсального субстрата» — и оно есть.
2. **Система переменных — образец data-driven.** Кость монаха d6→d8 как два разных
   эффекта на 1 и 5 уровнях — это **буквально парадигма №4** в коде. Мультикласс
   бесплатно. Мягкая деградация: ссылка на отсутствующую переменную не роняет лист.
3. **Формульный движок с инъекцией RNG** → детерминизм и тестируемость. Реальное
   покрытие: 70 тестов, lint-гейт над 244 механиками, свип-тест прод-контента.
4. **Корректные примитивы разрешения 2024 там, где реализованы.** Преимущество/помеха
   не складываются и взаимно гасятся (RAW 2024); temp HP берёт максимум, не сумму;
   концентрация — одна за раз; состояния маппятся в модификаторы 2024.
5. **`ValueBreakdown.parts` (source/reason)** — настоящее «превью из вычисления» для КЗ.
   Это правильный зародыш парадигмы №2 (объяснимое значение), просто пока только для КЗ.
6. **Чистое разделение** презентации и исполнения; чистый резолвер без React/глобалов,
   сериализуемые снапшоты `rule_state` (кэш, пересчитывается).
7. **Блочный конструктор** (trigger-блоки + effect-блоки) — авторинг данными без
   свободного текста. Сильный инструмент парадигмы №1.

## 4. Слабые стороны и фундаментальные пробелы (проверено, с файлами)

### 4.1. Нет шины событий / диспетчера триггеров — центральный пробел
`activation.mode:"triggered"` и весь словарь `event` (`hit`, `damage_taken`,
`attack_roll_made`, `creature_leaves_reach`, `reduced_to_0_hp`…) авторятся, но
исполняется **только** `long_rest` (`turn.ts` `applyLongRestPassives`). `EngineEvent`
(`events.ts`) — это **журнал** для UI, не pub/sub. Нет диспетчера, слушателей, подписок.

Следствия: не работают **реакции** (Щит, Адское возмездие, Контрзаклинание, Часовой,
Атака при возможности), **он-хит-райдеры** (Скрытая атака, Божественная кара, Внезапный
удар), **авто-проверка концентрации** при получении урона, срабатывания на движение/
0 HP. Это не косметика — без событий «единый движок в 2D-бою» нереализуем данными.

### 4.2. Обстоятельства (`when`/`circumstances`) не вычисляются
`collectFromPayload` (`modifiers.ts`) проверяет только `applies_to.roll`/`filter`,
никогда `payload.when`. Во всём `frontend/src/engine` — **ноль** ссылок на
`when`/`circumstances`/`target_has_condition`/`wielding`/`ally_within`. Значит
условные пассивки («преимущество, пока цель распластана», «+1 КЗ, пока щит в руках»,
Безрассудная атака только по `IsWeaponAttack`) либо применяются безусловно, либо не
применяются вовсе. Целая ветка §5.5 схемы — мёртвый груз в рантайме.

### 4.3. Парадигма №3 (методы расчёта) не реализована
`computeAC` — жёсткий каскад: `armor` → иначе первый `set_value ac_base` → иначе 10+dex.
`findAcBaseOverride` возвращает **первое** совпадение, не максимум. Ни `Math.max` по
кандидатам, ни реестра методов в коде нет. Персонаж с двумя безбронными методами
(Варвар 10+ЛВК+ТЕЛ и Доспех мага 13+ЛВК) **не получит больший**. А по RAW 2024:
> «If a rule gives you another base AC calculation, you choose which calculation to use;
> you can't use more than one.»
То есть база КЗ — это именно «набор кандидатов, взаимоисключающие, берём один (лучший
применимый)», плюс аддитивные бонусы поверх. Флагманский пример парадигмы №3 сам её не
исполняет.

### 4.4. Два расходящихся пути модификаторов
- **Боевой путь** (`modifiers.ts`): `Number(raw)` — берёт только литеральные числа;
  формульные модификаторы (`rage_bonus`, `cha_min_1`, `+prof`) → `NaN` → **молча
  отбрасываются**.
- **Путь листа** (`breakdown.ts` `modifiersFromEffects`): **formula-aware**, зовёт
  `evaluate()`.

Один и тот же модификатор по-разному учитывается на листе и в бою. Это их же риск R6
(«formula parsers diverge»), подтверждённый в коде. Ярость (+`rage_bonus` к урону)
в боевом броске просто исчезает.

### 4.5. Движок односторонний; сопротивления и урон не применяются
В `execute.ts` ветка `damage` считает число и пишет `damageEvent`, но **не трогает HP**
и не применяет сопротивления; `resistance` проваливается в `NOT_IMPLEMENTED`. HP меняется
только ручным `hp.ts applyDamage` из UI-панели. Причина системная: движок считает броски
**владельца листа**, но не моделирует **вторую сторону**. Поэтому:
- сопротивление B/P/S от Ярости в исполнении косметическое;
- нельзя выразить «атакующий бьёт с преимуществом, потому что цель распластана/опутана/
  ослеплена» — асимметрия состояний (§4.6) не проецируется на чужой бросок;
- 2D-бой требует контекста `attacker + target`, которого нет.

### 4.6. Состояния захардкожены (нарушение парадигмы №1)
`CONDITION_RULES` — TS-карта 13 состояний. Владелец контента **не может добавить**
состояние (хоумбрю, уровни Истощения 2024) без перевыкатки. Тест нарушения парадигмы №1
(«сможет ли владелец изменить без перевыкатки?») здесь проваливается. Асимметричные части
правил — только `note`-текст, не механика.

### 4.7. Превью из свободного текста — дрейф с механикой (парадигма №2 частично)
Ховер действия показывает `description` (свободный текст), а не гуманизированную
`mechanics`. `summarizeMechanics` работает из состояния блочного редактора, а не из
произвольного сохранённого `mechanics` (у AI-сгенерированной или руками-правленной
механики сводки нет). Единого `describe(mechanics) → текст`, общего для конструктора,
листа, disabled-состояния и боевого лога, **нет**. Значит текст, который видит игрок,
может расходиться с тем, что исполняется. Парадигма №2 выполнена лишь для значений
(breakdown КЗ), но не для механик.

### 4.8. Стекинг и «одноимённое не складывается» не смоделированы
Числовые модификаторы просто суммируются. Нет модели RAW 2024 «Combining Game Effects»:
> «when two or more game features have the same name, only the effects of one of them —
> the most potent one — apply while the durations overlap».
Нет `sourceName`-группировки, «берём максимум потенции», tie-break по свежести, продления
длительности при перекасте. Нет `StackId` (как в BG3). Нет авто-проверки концентрации
(`DC = clamp(max(10, ⌊dmg/2⌋),10,30)`, помеха при крите, потолок 30 — всё 2024).

### 4.9. Прочее (известное / RAW-долги)
- **Отдыхи упрощены:** `shortRest` лечит 50% макс.HP безусловно — не по RAW (2024:
  трата Костей Хитов). Модели Костей Хитов нет.
- **Не реализованы payload'ы:** `boon`, `reroll`, `set_die`, `transform`, `grant_action`,
  `movement`-в-цель, `resistance`-в-урон — narrative/заглушки.
- **grant-семейство / pending-choice / upcast-UI** — авторятся, но рантайм-разрешение
  выбора и апкаст-пикер отложены.
- **Оружейное мастерство 2024** (8 свойств: Cleave/Graze/Nick/Push/Sap/Slow/Topple/Vex) —
  чисто он-хит/он-мисс подсистема, которой некуда «прицепиться» без событий (§4.1).

## 5. Уроки Baldur's Gate 3 (модель данных, которую стоит перенять)

BG3 реализует ровно ваши идеи, но с явной послойной моделью данных. Ниже — **дословные**
определения из стат-данных BG3, каждое иллюстрирует недостающий у вас слой.

**Слой 1 — Boost (атомарная дельта стата, декларативно, с условием).**
```
AgonizingBlast:
  Boosts: IF(SpellId('Projectile_EldritchBlast')):DamageBonus(CharismaModifier, Force)
  StatsFunctorContext: OnAttack
  Conditions: SpellId('Projectile_EldritchBlast') and HasDamageEffectFlag(DamageFlags.Hit)
```
Это **условный модификатор как данные** — ваш `modifier.when`, только у BG3 он реально
вычисляется через `IF(...)`-грамматику. Boost'ы: `AC(2)`, `RollBonus(Attack, X)`,
`Advantage(AttackRoll)`, `Resistance(Fire, Resistant)`, `Ability(Strength, 2)`,
`WeaponDamage(1d4, Fire)`. Прямая аналогия вашему `Payload.modifier` — но с `IF()`.

**Слой 2 — Functor (императивное «сделай», в контексте события).**
`ApplyStatus(...)`, `DealDamage(1d4, Fire)`, `RemoveStatus(...)`, `RegainHitPoints(...)`.
Ваш `applyPayloads` — это функторы. Отличие: у BG3 они висят на **контекстах**
(`OnApplyFunctors`, `TickFunctors`, `OnRemoveFunctors`, `StatsFunctors`+`Context`).

**Слой 3 — Status (контейнер = бусты + тик + жизненный цикл + идентичность стека).**
```
RAGE:
  StatusType: BOOST
  StackId: RAGE
  TickType: EndTurn
  Passives: Rage_Rage_Boosts; Rage_Attack; Rage_Damaged; Rage_NoHeavyArmour_VFX
  RemoveEvents: OnTurn; OnCombatEnded
  OnApplyFunctors:  IF(not HasHeavyArmor(context.Source)):ApplyStatus(RAGE_NO_HEAVY_ARMOUR_VFX,...)
  OnRemoveFunctors: IF(RemoveCause(StatusRemoveCause.TimeOut)):UseSpell(Shout_EndRage,...)
```
Ключевое: **статус ссылается на несколько Passive** — гранулярная композиция (ваша
парадигма №4!). `StackId` даёт «одноимённое не складывается» из коробки (ваш §4.8).
`RemoveCause` позволяет ветвить снятие (истёк / развеян / прерван). `BURNING` аналогично:
`TickType: StartTurn`, `TickFunctors: IF(not Tagged('TORCH')):DealDamage(1d4,Fire)`.

**Слой 4 — Interrupt (реакция как чистые данные — ваш пробел №4.1).**
```
Interrupt_Sentinel:
  InterruptContext: OnCastHit
  InterruptContextScope: Nearby
  Container: YesNoDecision
  Conditions: Ally(context.Target,context.Observer) and Enemy(context.Source,context.Observer)
              and IsMeleeAttack() and not DistanceToEntityGreaterThan(2, ...)
              and IsAbleToReact(context.Observer) and HasLastAttackTriggered()
  Cost: ReactionActionPoint:1
  InterruptDefaultValue: Ask; Enabled
  Properties: UseAttack(OBSERVER_SOURCE)
```
```
Interrupt_MAG_Counterspell:
  InterruptContext: OnSpellCast
  Roll:    TryCounterspellHigherLevel(3)
  Success: UseSpell(OBSERVER_SOURCE, Target_MAG_CounterSpell_Success, ...)
  Failure: UseSpell(OBSERVER_SOURCE, Target_Counterspell_Failure, ...)
  Cost: ReactionActionPoint:1; Interrupt_MAG_Counterspell:1
  Cooldown: OncePerRestPerItem
```
Реакция = `{контекст-события, область, условия-предикат, стоимость (реакция), timing,
Ask/Enabled, [Roll → Success/Failure]}`. Это в точности «event bus, который правила
только подразумевают».

**Что перенять (таблица соответствий):**

| BG3 | У вас сейчас | Что делать |
|---|---|---|
| Boost с `IF(cond):...` | `Payload.modifier` без вычисления `when` | вычислять `when` в едином конвейере (§7.C) |
| Functor + Context | `applyPayloads` без контекстов | привязать функторы к событиям (§7.A) |
| Status.`Passives:[...]` | активный эффект = один payload/механика | статус-контейнер, ссылающийся на набор эффектов (парадигма №4) |
| `StackId` / most-potent | суммирование | `StackId` + «одноимённое = максимум потенции» (§7.D) |
| Interrupt (реакция-данные) | нет | диспетчер реакций (§7.A) |
| `Roll → Success/Failure` в реакции | нет | ветвление реакции (Контрзаклинание) |

### 5.1. Стекинг — три поля данных (готовое решение вашего §4.8)
BG3 решает «заменить или сложить?» декларативно, без кода, тремя полями статуса:
- **`StackId`** — строковый ключ идентичности. Два статуса с одинаковым `StackId`
  не сосуществуют — коллизия и запускает логику стека. (Это и есть RAW 2024
  «одноимённое не складывается».)
- **`StackType`** — политика при коллизии: `Overwrite` (заменить), `Ignore`
  (оставить старое, отбросить новое), `Additive` (сложить длительности:
  Излучающий Ореол 1+2=3 хода), `Stack` (независимые экземпляры — истинный мульти-стек).
- **`StackPriority`** — тай-брейк при равном `StackId` (у temp HP приоритет 1: любой
  новый источник ≥1 перекрывает независимо от величины).

Перенос: добавить `stack_id`/`stack_type`/`stack_priority` на активный эффект. Это
покрывает почти все вопросы D&D «заменяет или складывает» данными.

### 5.2. Ключевой структурный факт: контексты-триггеры — битфлаги
Внутренний enum `StatsPropertyContext` — **степени двойки** (`CAST=4, ATTACK=32768,
DAMAGED=4194304, LONG_REST=67108864…`). Значит `StatsFunctorContext:
OnCast;OnStatusRemoved;OnStatusApplied` — это **OR-набор**: функторы срабатывают, если
случилось **любое** из перечисленных событий. Пары active/passive (`OnAttack`/`OnAttacked`,
`OnDamage`/`OnDamaged`) веером раздают одно событие обеим сторонам. Проектируя свою шину
(§7.A), моделируйте подписку эффекта как OR-множество типов событий — дёшево и расширяемо.

### 5.3. Идиома «буст-при-условии = переключение статуса»
Вместо буста, читающего живое состояние, BG3 держит **бусты статичными**, а всю
динамику — в триггерном слое: пассивка слушает события (`OnStatusApplied/Removed`,
`OnDamaged`…) и **навешивает/снимает** вспомогательный статус, чьи `Boosts` и есть
реальная дельта. Это альтернатива «вычислять `when` при каждом броске» (§7.C): условные
пассивки можно реализовать как правила, материализующие/снимающие эффект по событию.
Обе стратегии валидны; вторая держит горячий путь бросков тривиальным.

### 5.4. Точные совпадения с вашими парадигмами
- **Пояс силы огра (парадигма №3).** У BG3 для этого есть буст
  `AbilityOverrideMinimum(Strength,15)` — ровно ваш «метод СИЛ как ещё один кандидат».
- **Защита без доспехов.** `ACOverrideFormula(...)` целиком заменяет формулу КЗ (ваш
  `set_value ac_base`), а тонкости КЗ разнесены на гранулярные бусты
  (`AddProficiencyToAC`, `ArmorAbilityModifierCapOverride`, `BlockAbilityModifierFromAC`) —
  это парадигма №4 (дробить максимально) в чистом виде.
- **Реакция спрашивает игрока (парадигма №2).** У интеррапта есть контракт
  `InterruptDefaultValue: Automatic|Ask|Disabled` + `Container: YesNoDecision`. Причём
  «нельзя применить две реакции к одному событию, если хоть одна с подтверждением» —
  готовая модель UX реакций.
- **Полнота словарей.** Definitive-списки (перепроверяемо): ~160 бустов (`BoostType`),
  ~65 функторов (`StatsFunctorId`) — см. §10. Полезно как чек-лист покрытия вашего
  `Payload.kind`.

## 6. Сверка с 4 парадигмами

| Парадигма | Оценка | Комментарий |
|---|---|---|
| №1 — Без хардкода | 🟡 частично | Ось эффект/действие/заклинание — да (data-driven). Но **состояния** (§4.6), **методы КЗ** (§4.3), turn-ресурсы — хардкод. Тест нарушения проваливается на состояниях. |
| №2 — Превью везде | 🟡 частично | Значения (breakdown КЗ) — да. Механики действий/эффектов — нет: ховер из свободного текста, дрейфует; нет общего `describe(mechanics)` (§4.7). «Причина недоступности слоем поверх» есть в панели действий, но поверх текста, не поверх гуманизированной механики. |
| №3 — Методы расчёта | 🔴 не реализовано | Есть только для КЗ и то как каскад «первый», а не «максимум кандидатов» (§4.3). Нет обобщённого реестра методов. |
| №4 — Гранулярность эффектов | 🟢 да | Система переменных — эталон: d6→d8 как разные эффекты по уровням; мультикласс; источник виден. Именно это BG3 делает через `Status.Passives:[...]`. |
| Сквозное: движок-субстрат | 🟡 частично | Один интерпретатор — да. Но без шины событий (§4.1) и двусторонности (§4.5) он не дотягивает до «создание → лист → 2D-бой → кампания». |

## 7. Фундаментальные изменения (по приоритету, конкретно)

### A. Шина событий + диспетчер триггеров/реакций (наивысший рычаг)
Ввести явную событийную модель — тот самый субстрат, который правила лишь
подразумевают, а BG3 сделал данными (Interrupt).

- **Событие:** `{type, source, target?, payload, timing-phase}`. Типы из вашего же
  словаря: `attack_roll_made`, `hit`, `crit`, `damage_taken`, `reduced_to_0_hp`,
  `creature_leaves_reach`, `spell_cast`, `turn_start/end`, `save_made`…
- **Диспетчер:** при исполнении действия движок эмитит события в фазах
  `before → during → after → replaces`; для каждого собирает подходящие
  `triggered`/`reaction`-механики (по `event` + `subject` + вычисленным `circumstances`).
- **Реакции:** `reaction` тратит `reaction`-ресурс, поднимает выбор игрока
  (`Ask`/`Enabled` как в BG3 `InterruptDefaultValue`), поддерживает
  `Roll → on_success/on_fail` (Контрзаклинание). `triggered` без реакции —
  авто (Скрытая атака, Неумолимая стойкость).
- **Что моментально включается данными:** Щит, Адское возмездие, Часовой, ОА,
  Скрытая атака, Божественная кара, Внезапный удар, авто-проверка концентрации при
  `damage_taken`, оружейное мастерство (он-хит/он-мисс).
- **Подписка = OR-набор типов событий** (структурный урок BG3, §5.2): эффект слушает
  «любое из {damage_taken, attack_roll_made,…}». Пары active/passive
  (`attack`/`attacked`) раздают одно событие обеим сторонам (нужно для §E).
- **UX реакции = парадигма №2:** политика `Automatic|Ask|Disabled` на реакцию,
  Yes/No-подсказка (как `InterruptDefaultValue`/`Container` в BG3), правило «не больше
  одной подтверждаемой реакции на событие».
- Реализация: единый `dispatch(event, state, ctx)` рядом с `execute.ts`; таблица
  слушателей строится из `passives` + `activeEffects` (у которых `activation.mode ∈
  {triggered, reaction}`). Timing по вашему `trigger.timing`.

### B. Настоящие «методы расчёта» (парадигма №3 как механизм, не спецкод для КЗ)
Обобщить `computeAC` в общий примитив:
```
derived(value) = max{ m.eval(state) | m ∈ methods(value), m.requirements(state) all pass }
                 + Σ aadditive_bonuses (с дедупом по source-name)
```
- Метод-кандидат — это payload `{kind:"value_method", target:"ac_base",
  formula:"10+dex+con", requirements:[armor:none]}`. Защита без доспехов, Доспех мага,
  Драконья стойкость, брони — все становятся кандидатами; движок берёт максимум
  применимого (RAW 2024: «you choose… can't use more than one»).
- Тот же механизм — для СЛ спасбросков, инициативы, скорости, **и характеристик**
  (Пояс силы огра как метод СИЛ — конкурирует с базовым; в BG3 это готовый буст
  `AbilityOverrideMinimum(Strength,15)`, §5.4).
- `ValueBreakdown` уже несёт `parts` — расширить его «почему выбран этот метод и какие
  были кандидаты» (кормит парадигму №2).

### C. Единый formula-aware конвейер модификаторов + вычисление `when`
- Слить боевой (`modifiers.ts`) и листовой (`breakdown.ts`) пути в **один**
  formula-aware сборщик. Убить `Number(raw)`-путь (§4.4).
- В сборщике **вычислять `when`/`circumstances`** через маленький предикатный
  интерпретатор (аналог BG3 `Conditions`: `HasStatus`, `Tagged`, `IsWeaponAttack`,
  `target_has_condition`, `ally_within`). Это оживит §4.2 и слой Boost из §5.

### D. Состояния и стекинг как данные
- **Сущность Condition** (как effect/action): `self_modifiers` + `projected_modifiers`
  (что даёт атакующему/цели рядом) + `notes`. Тогда владелец добавляет состояние без
  перевыкатки (парадигма №1), а асимметрия (преимущество по распластанному) становится
  исполнимой (нужен §E).
- **Стекинг по RAW 2024 = три поля данных** (модель BG3, §5.1): `stack_id` (идентичность),
  `stack_type ∈ {overwrite, ignore, additive, stack}` (политика коллизии), `stack_priority`
  (тай-брейк). «Одноимённое — максимум потенции, свежесть решает ничью, длительность
  продлевается» ложится на `overwrite`+priority. Числовые бонусы разных имён складываются;
  типов бонусов у D&D нет — модель = строковая идентичность + набор флагов «не
  складывается» (укрытие, тип магопредмета, методы КЗ).
- **Авто-концентрация:** на `damage_taken` для концентрирующего кидать
  `DC = clamp(max(10,⌊dmg/2⌋),10,30)`, помеха при крите; перекаст концентрации рвёт
  старую.

### E. Двусторонний контекст боя
- `ExecuteContext` должен нести **rule-state атакующего и цели**, чтобы состояния/эффекты
  цели меняли бросок атакующего (преимущество по распластанному/опутанному/ослеплённому,
  укрытие), а урон применялся к HP цели **с учётом сопротивлений/иммунитета/уязвимости**
  (реализовать §4.5). Это делает движок пригодным для 2D-боя без нового кода — данными.
- Урон: ветку `damage` довести до применения к HP цели через общий `applyDamage`
  (сейчас он в UI) с проходом сопротивлений.

### F. Общий `describe(mechanics) → текст` (парадигма №2 везде)
- Один чистый humanizer, принимающий **сохранённый `mechanics`** (не состояние
  редактора), используемый конструктором (сводка), листом (ховер), disabled-состоянием
  и боевым логом. Причина недоступности («Недостаточно ресурсов») — слой **поверх**, как
  и требует парадигма №2, но поверх гуманизированной механики, а не вместо неё.
- Это же убирает дрейф текст↔механика (§4.7) и даёт превью AI-сгенерированным механикам.

## 8. Кардинальные изменения (стоит держать в голове)

1. **Event-sourced `RuleState` reducer.** У вас уже есть `CharacterEvent` (персистентные
   события) и чистый резолвер. Следующий шаг — сделать персонажа **проекцией потока
   событий**: создание, левелап, экипировка, каст, урон, отдых — всё события; лист, бой и
   кампания — разные проекции одного редьюсера. Это буквальная реализация «единого
   интерпретатора на всём пути» и естественно вбирает шину событий (§7.A). Побочно:
   time-travel/undo, реплей боя, серверная валидация.
2. **Движок как общий TS-пакет.** Вынести `engine/*` в изолированный пакет без React,
   импортируемый (а) клиентом-листом, (б) серверной автоматизацией боя (`n8n_battle_
   automation_plan.md`), (в) будущей кампанией. Один интерпретатор физически на всём пути,
   а не только концептуально. Требует чистых границ (уже почти есть: инъекция RNG,
   сериализуемые снапшоты).
3. **Явное разделение Boost / Functor (по образцу BG3).** Сейчас `modifier` (декларативная
   дельта) и `damage/heal/apply` (императив) живут вперемешку в `applyPayloads`. Явно
   разделить «пассивные бусты, влияющие на вычисление значений» и «функторы, меняющие
   состояние по событию» — упростит и §7.B, и §7.C, и §7.D.

## 9. Порядок внедрения (предложение)

1. **C** (единый formula-aware конвейер + `when`) — дешёвый, чинит тихие баги, разблокирует условные пассивки.
2. **B** (методы расчёта) — средний, закрывает парадигму №3, переиспользует `ValueBreakdown`.
3. **A** (шина событий + реакции) — крупный, но наивысший рычаг; после него десятки способностей включаются данными.
4. **D** (состояния/стекинг данными) — опирается на A/C.
5. **E** (двусторонний бой) — опирается на A/D; открывает 2D-бой.
6. **F** (общий humanizer) — можно параллельно; завершает парадигму №2.
7. Кардинальные §8 — по мере готовности; §8.1 естественно вырастает из A.
