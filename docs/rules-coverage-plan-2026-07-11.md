# План 100% покрытия правил PHB 2024 — классы/подклассы/виды/подвиды (2026-07-11)

Составлен по веерной разведке (6 читателей по подсистемам + критик полноты, ~700k
токенов, все факты с пруфами `файл:строка`), затем прошёл адверсариальное ревью
3 независимых критиков (соответствие задаче / реализуемость по коду / жизнеспособность
процесса) — все подтверждённые находки внесены. Цель — для каждого класса, подкласса,
вида и подвида: (1) убедиться, что система выдаёт ВСЕ фичи из правил; (2) каждую фичу
отнести к категории поддержки; (3) исполнимое закрепить тестами на живых данных;
(4) поддерживать это состояние постоянно.

## 2. Таксономия категорий (по требованиям владельца)

Категория присваивается **аспекту** фичи; фича целиком получает свёртку своих аспектов.
Пример: Ярость = ресурс (полн.) + бонус урона (полн.) + сопротивления (полн.) +
«не можешь читать заклинания» (доработка/нарратив) → фича «частично».

| Код | Категория | Критерий | Следствие |
|---|---|---|---|
| `full` | Полностью поддерживается | все механические аспекты выражены payload'ами; исполняются без NOT_IMPLEMENTED/инертных предикатов; числа = канону | **обязателен** golden-тест на живых данных |
| `partial` | Частично | есть исполнимая часть и нарративная | тест на исполнимую часть + фиксация нарративной |
| `needs_engine` | Поддерживается с доработками | правило выразимо, но упирается в дыру интерпретатора (§6) | описание доработки + ссылка на пункт §6; после доработки → full |
| `narrative` | Не может быть поддержано | текст чисто повествовательный (ролевые/социальные права, разрешения мастера) | фиксируется в реестре с обоснованием; НЕ считается провалом |

**Автодетекторы дают кандидата категории, финальное слово — за вычиткой правила:**
- `mechanics == null` или только `kind:'narrative'` → кандидат narrative/needs_engine
  (нужен новый детектор «фича выдаёт 0 payload'ов, влияющих на state» — сейчас такие
  фичи невидимы: линт их пропускает, exec-свип не находит NOT_IMPLEMENTED, т.к. нечего исполнять);
- исполнение с событием `NOT_IMPLEMENTED payload/resolution`, инертным предикатом
  (predicates.ts:44-53) или PLANNED-событием шины → кандидат needs_engine;
- гранты применились + payload'ы исполнились → кандидат full (но full присваивается
  только после сверки ЧИСЕЛ с каноном — схемная валидность ≠ правильность).

## 3. Архитектура сверки: канонический реестр — ось всего

Следуя парадигме №1 («моделируй данными, не кодом»), покрытие — это **данные**:
канонический реестр фич + универсальный тест-раннер, интерпретирующий декларативные
проверки. Не 60 рукописных тест-файлов, а один раннер × 60 JSON-юнитов.

### 3.1 Реестр: `officials/canon/phb2024/<unit>.json` (в git)

  "names": { "ru": "Варвар", "en": "Barbarian" },
  "source": { "ru_pages": [61, 68], "en_pages": [49, 56] },  // печатные страницы kb
  "match": { "class_name": "Варвар" },                        // привязка к прод-сущности

    "hit_die": "d12",

Обязательные свойства формата (уроки критика):
- **пара имён RU+EN + card_number/slug** — матчинг только по имени ненадёжен
  (гомоглифы, ё, легаси-дубли, расхождения перевода);
- **числовые таблицы вводятся вручную** с полем `verified_by` — автоизвлечению из kb
  не доверять (подтверждённые повреждения в обоих слоях);
- реестр ссылается на печатные страницы kb — при перегенерации базы ссылки живы.

### 3.2 Конвейер наполнения реестра (воркфлоу агентов — паттерн уже обкатан
на 48 подклассах и 393 заклинаниях)

1. **Экстрактор** (на юнит): читает kb RU-страницы юнита → черновик реестра
   (фичи по уровням, тексты, аспекты).
2. **Верификатор**: сверяет с EN-страницами (имена, уровни, состав) + флажит числа.
3. **Классификатор**: каждой фиче — категория; сверяет с матрицей движка (§6) и
   прод-механикой фичи; описывает доработку для needs_engine.
4. **Адверсариальный критик**: пытается опровергнуть категоризацию (особенно full —
   «а числа проверены? а на листе, не только в бою?»).
5. Ручная верификация числовых таблиц (владелец или отдельный проход по печатному PDF).

### 3.3 Инструмент сверки: аудитор юнитов

**Форма исполнения:** НЕ bare-Node `.mjs` — конвейер кузницы тянет TS-модули фронтенда с
`import.meta.env` (api/client.ts:55) и требует шим localStorage. Поэтому: `autoBuild()`
выносится из forge.sweep в общий TS-модуль, аудитор гоняется под vitest/vite-node
(гейт-«тест», который пишет отчёты в `docs/coverage/`).

**Источник тел сущностей:** git-снапшот (0.3), живой API — только fallback/smoke.
Живой прод не выдержит полный аудит: `loadBundle` — N+1 GET на тела эффектов при каждой
пересборке, autoBuild — до 6 проходов; ~99 юнитов × 20 уровней ≈ 2000 сборок = часы и
тысячи запросов к Railway. Нужен офлайн-резолвер (fetch-стаб над снапшотом) + кэш тел
между уровнями юнита.

Для каждого юнита реестра, на уровнях 1..20:
1. собрать персонажа **реальным конвейером кузницы** (расширенный `autoBuild()`:
   + параметр уровня, + выбор подкласса, + ASI/черты, + стратегия выбора заклинаний);
2. перечислить всё выданное: сущности сборки, appliedGrants (несут source — трассируемость
   встроена), ресурсы, переменные, pendingChoices;
3. сматчить с реестром: **missing** (в каноне есть, система не выдала), **extra**
   (система выдала, в каноне нет и нет в extra_allowlist юнита), **inert** (выдано,
   но 0 state-payload'ов);
4. отчёт JSON+MD на юнит: `docs/coverage/<unit>.md` — это и есть «отчёт по каждому»,
   который требует владелец.

Метрика покрытия юнита:
`покрытие = (фич классифицировано ∧ full/partial закрыты checks-проверками) / всего фич канона`.
100% = каждая фича канона классифицирована И всё исполнимое под тестом. narrative
не понижает покрытие — понижает только «не классифицировано» и «full без теста».
**Отдельно от метрики: missing ≠ 0, extra ≠ 0 (вне allowlist) и inert ≠ 0 — каждый
сам по себе красный сигнал аудита**, иначе легаси-дубли и лишние гранты пройдут незамеченными.

### 3.4 Канонический тест-раннер: `frontend/src/canon/canon.test.ts`

Один data-driven раннер (расширение паттерна `contentSweep.test.ts`, где JSON —
источник правды и для теста, и для сида):
- итерирует `officials/canon/phb2024/*.json`;
- `checks.type='grant'/'resource'/'variable'/'choice_offered'` — через мини-сборку
  `resolveCharacterRules` или живой `autoBuild` (гейт `MVP_CONTENT=1`);
- `checks.type='stat'` — core_traits класса (hit_die/спасброски/владения/skill_choices/
  снаряжение) и поля вида (speed/darkvision/size) против полей прод-сущности;
- `checks.type='spell_list'` — канонический список заклинаний класса против членства
  `spell.classes`/`spell.subclasses` в прод-БД (+ always-prepared подклассов);
- `checks.type='value'` — golden-числа листа через `breakdownValue(key, ...)`
  (образец: fightingstyle.forge.mvp.test.ts:98-119);
- `checks.type='execute'` — исполнение с seeded rng, ожидаемые события/состояние
  (образцы: mechanics.sweep richState/stripCost, execute.mvp);
- незакрытые checks у full/partial-фич = провал теста.

Три оси проверки обязательны для full: **выдача** (сборка), **числа** (лист),
**исполнение** (рантайм). Схемная валидность ≠ исполняемость ≠ правильность чисел —
сейчас в проекте меряется в основном первая ось.

### Этап 1 — Пилот: Варвар (класс + 4 подкласса; ~2-3 дня)

Пилот проверяет ВЕСЬ конвейер на классе с известными дырами (LP обрезан на 8 уровнях,

3. прогнать `audit-units.mjs` → отчёт missing/extra/inert;
4. классифицировать все фичи 1–20 + 4 подкласса (3/6/10/14);
5. закрыть missing контентом (дозаполнить LP 9–20), написать checks, добиться зелёного

6. зафиксировать выявленные слабости формата реестра/раннера, обновить шаблон.

**Выход пилота — калиброванная оценка объёма** на оставшиеся 55 класс-юнитов и 39 видовых.

## 5. Постоянное поддержание зелёного состояния

1. **CI-гейты** (этап 0.4): PR — оффлайн-наборы; ночь — canon-аудит и раннер ПО СНАПШОТУ
   (быстро, стабильно) + тонкий живой smoke по проду (forge-свип L1, mechanics-свип,
   lint-mechanics) + дифф снапшота. Любое расхождение — красный прогон с отчётом
   (какой юнит, какая фича, какая ось).
2. **Правило контента**: новый/изменённый контент попадает в прод только через
   сид-скрипт + запись в реестре + checks (паттерн contentSweep.fixtures.json: один JSON
   читают и тест, и сид). Ручная правка в UI прод-БД ловится ночным диффом снапшота.
3. **Процедура красного диффа снапшота** (правка в UI — живая практика, конструкторы
   только что доделаны, так будет часто): (а) **allowlist нечувствительных полей**
   (image_url, description-тексты и пр. — enrich/икон-скрипты массово их меняют, иначе
   дифф красный каждую ночь); (б) чувствительный дифф (mechanics/LP/related_*/resources) →
   авто-PR «обновить снапшот» с прогоном canon-раннера по новому срезу: раннер зелёный →
   владелец мержит (правка легитимна), красный → правка сломала покрытие, откатить
   контент или обновить реестр осознанно. Триаж — владелец (или дежурная модель).
4. **Правило движка**: расширение схемы = синхронно расширить интерпретатор + контракты
   (HANDLED/EMITTED) + AI-промпт (ai_mechanics_controller.go) + перекатить canon-раннер;
   контракт-тесты (eventBusContract, validateMechanics.test) держат схему↔движок.
5. **Реестр — ЕДИНСТВЕННЫЙ источник правды покрытия**: статусы фич меняются только
   с обоснованием (закрыл needs_engine → перегнать классификатор/тест).
   `scripts/coverage/report.mjs` печатает % покрытия по юнитам. Существующие
   параллельные «источники правды» сводятся к реестру: `scripts/content/mvp-requirements.mjs`
   (меряет только уровни 1–2, круги 0–1) — генерируется из реестра или деприкейтится;
   `contentSweep.fixtures.json` — вливается в реестр как checks соответствующих фич.
6. **Дашборд**: сводный `docs/coverage/README.md` (генерируется) — таблица юнитов,
   % покрытия, счётчики категорий (full/partial/needs_engine/narrative + missing/extra/inert),
   дата последнего зелёного живого прогона.
